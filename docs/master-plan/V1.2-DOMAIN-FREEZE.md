# V1.2 — K1 — Domain Model Freeze & Architecture Kick-off

**Status:** K1 — Frozen (amended by K2 2026-07-01)  
**Date:** 2026-07-01  
**Precondition:** V1.2 K0 Vision & Scope Frozen  
**Supersedes:** Nothing (new document)

> **K2 Amendments applied:**
> - A-1: ObservationStore is an independent Aggregate Root (not owned by CandidateProfile)
> - A-2: Canonical flow is EvidenceSignal → ObservationExtractor → Observation (not Evaluation → Observation directly)
> - A-3: ReasonerDecision is explicit in the canonical runtime flow
> - A-4: FeatureExtractor renamed and elevated to FeatureEngine (orchestrates Updaters)
> - A-5: ProgrammingLanguage is an abstract domain concept; Python/JS/TS/etc. are concrete implementations

---

## SECTION A — Domain Vision

### Conceptual Evolution from V1.1

V1.1 established **Interview Reasoning** — the ability to observe candidate behaviour, detect patterns, and build a `CandidateProfile` within a single session. The pipeline was fully built but its outputs stopped at the Reasoner. Nothing downstream consumed the profile.

V1.2 completes the pipeline in four conceptual stages:

```
Interview Reasoning      (V1.1 — complete)
       ↓
Candidate Knowledge      (V1.2 — EPIC-01 / EPIC-02)
  The CandidateProfile becomes a live, consuming object.
  ProfileFeatures are readable by NarrativeGenerator and ReportBuilder.
  ObservationStore formalises the temporal fact layer.
       ↓
Learning Intelligence    (V1.2 — EPIC-04 / EPIC-05)
  Gap signals are converted into ranked study recommendations.
  Evidence Freshness ensures the profile reflects recent performance.
  The platform knows not only what the candidate did, but what they should do next.
       ↓
Continuous Coaching      (V1.2 — EPIC-03 / EPIC-06 / EPIC-07)
  Reports become personalised coaching documents, not generic score sheets.
  Calibration guarantees scoring integrity as the platform scales.
  Enterprise Extensibility opens the door to longitudinal tracking across organisations.
```

### New Conceptual Boundaries in V1.2

| Boundary | V1.1 | V1.2 |
|---|---|---|
| CandidateProfile | Built by detectors; never consumed downstream | Consumed by NarrativeGenerator and ReportBuilder |
| Observation | Implicit, inside detector logic | First-class domain concept with lifecycle |
| Evidence | Written to EvidenceStore per detector | Elevated to ObservationStore with decay semantics |
| Narrative | Generic coaching text per dimension | Profile-feature-aware, evidence-anchored sections |
| Coaching | Not a distinct concept | Formalised as CoachingPlan with ranked CoachingActions |
| Language | An execution config flag | First-class domain concept: ProgrammingLanguage with LanguagePolicy |
| Session History | Not persisted | Durable SessionHistory with replay capability |
| Calibration | Implicit in constants | Explicit CalibrationProfile with CI validation |

## SECTION B — Domain Concepts

### Observation

**Purpose:** An immutable temporal fact — a single, typed, timestamped record that something happened during a session (e.g. "candidate hesitated on edge cases", "strong reasoning depth signal detected").  
**Ownership:** ObservationStore  
**Lifecycle:** Created by detectors during evaluation; never mutated; expires after TTL (Evidence Freshness).  
**Producers:** All pattern detectors (10 in V1.1), EvaluationSignalDetector, evaluation pipeline.  
**Consumers:** FeatureUpdater, ObservationStore queries, CalibrationProfile.  
**Persistence:** Transient within session in V1.2; candidate for cross-session persistence in V1.3+.

---

### ObservationStore

**Purpose:** The ordered, append-only registry of all Observations for a session or profile scope. The single source of temporal facts for the profile layer. **Independent Aggregate Root (K2 amendment — not owned by CandidateProfile).**  
**Ownership:** Independent Aggregate Root; written exclusively by ObservationExtractor.  
**Lifecycle:** Initialised at session start; accumulates Observations throughout; applies decay function at each detector cycle; expires stale entries.  
**Producers:** ObservationExtractor (sole writer — transforms EvidenceSignals into Observations).  
**Consumers:** FeatureEngine, CalibrationProfile, ReplayUpdater.  
**Persistence:** Session-scoped in V1.2.

---

### ObservationType

**Purpose:** A controlled vocabulary of Observation categories, ensuring Observations are typed and comparable across sessions and detectors.  
**Ownership:** Domain contracts layer.  
**Lifecycle:** Static; extended only when a new detector or signal type is introduced.  
**Producers:** Domain design (not runtime).  
**Consumers:** All detectors, ObservationStore, FeatureExtractor.  
**Persistence:** Code artifact only (enum or registry).

---

### ProfileFeature

**Purpose:** A derived, named capability or behavioural trait synthesised from one or more Observations. Examples: `ConfidenceCalibration`, `AdaptabilityUnderPressure`, `CollaborativeReasoning`. Represents what the candidate is, not what they did.  
**Ownership:** CandidateProfile aggregate.  
**Lifecycle:** Derived (never directly authored); recalculated whenever the ObservationStore changes; version-controlled schema.  
**Producers:** FeatureExtractor (reads ObservationStore → produces ProfileFeatures).  
**Consumers:** NarrativeGenerator, ReportBuilder, CoachingPlan.  
**Persistence:** Derived at report time in V1.2; candidate for cross-session accumulation in V1.3+.

---

### FeatureUpdater

**Purpose:** Orchestrates the update of a specific ProfileFeature when new Observations arrive. Applies feature-specific aggregation logic (e.g. rolling confidence average, boolean activation threshold).  
**Ownership:** CandidateProfileEngine (services/interview_reasoner/profile/).  
**Lifecycle:** Invoked per detector cycle; stateless per invocation; reads ObservationStore.  
**Producers:** CandidateProfileEngine (calls FeatureUpdater per registered feature).  
**Consumers:** Produces updated ProfileFeature values; consumed by CandidateProfile.  
**Persistence:** None (pure computation).

---

### FeatureExtractor → FeatureEngine

**Purpose (K2 amended):** `FeatureEngine` orchestrates multiple Updaters and is the sole producer of all ProfileFeatures. Known Updaters: `ObservationUpdater`, `ReplayUpdater`, `CalibrationUpdater`, future `LearningUpdater`. Reads from ObservationStore; writes only to CandidateProfile.features.  
**Ownership:** Independent domain service.  
**Lifecycle:** Invoked per detector cycle and at session close; stateless per invocation.  
**Producers:** Domain design (composition of Updaters).  
**Consumers:** FeatureEngine output consumed by CandidateProfile.  
**Persistence:** None (pure computation).

---

### KnowledgeGap

**Purpose:** A named, severity-scored deficit in a specific knowledge area, derived from evaluation signals. Represents what the candidate does not yet know or cannot yet do.  
**Ownership:** KnowledgeGapEngine (services/).  
**Lifecycle:** Computed per session from EvaluationResults; ranked by severity; consumed by CoachingPlan and StudyRecommendation.  
**Producers:** KnowledgeGapEngine.  
**Consumers:** StudyRecommendation, CoachingPlan, ReportBuilder.  
**Persistence:** Persisted with SessionHistory in V1.2.

---

### StudyRecommendation

**Purpose:** A single, actionable learning directive — topic + gap description + 2–3 resources + estimated study time — mapped to a specific KnowledgeGap.  
**Ownership:** CoachingEngine (services/).  
**Lifecycle:** Generated per session from KnowledgeGap list; ranked by gap severity × IRS impact; included in report.  
**Producers:** CoachingEngine (reads KnowledgeGaps → maps to resource library → ranks).  
**Consumers:** ReportBuilder, candidate-facing report section.  
**Persistence:** Persisted with SessionHistory.

---

### LearningObjective

**Purpose:** A higher-level goal that groups one or more StudyRecommendations under a named theme (e.g. "Master async patterns in JavaScript"). Provides narrative coherence to the study plan.  
**Ownership:** CoachingPlan aggregate.  
**Lifecycle:** Derived from grouped StudyRecommendations; one per knowledge area cluster.  
**Producers:** CoachingEngine.  
**Consumers:** CoachingPlan, ReportBuilder.  
**Persistence:** Derived; stored with CoachingPlan in V1.2.

---

### NarrativeSection

**Purpose:** A named, structured block of coaching text within the session report. Each section corresponds to a reporting area (e.g. Reasoning Quality, Communication, Knowledge Gaps). In V1.2, sections are profile-feature-aware.  
**Ownership:** NarrativeGenerator.  
**Lifecycle:** Generated at report assembly time; structured output schema enforced; references specific ProfileFeatures or KnowledgeGaps.  
**Producers:** NarrativeGenerator (reads CandidateProfile + KnowledgeGaps).  
**Consumers:** ReportBuilder.  
**Persistence:** Stored as part of SessionHistory.

---

### NarrativeInsight

**Purpose:** A single, specific coaching observation within a NarrativeSection — the atomic unit of personalised feedback. Must reference at least one Observation or ProfileFeature as its evidence anchor.  
**Ownership:** NarrativeGenerator.  
**Lifecycle:** Generated alongside NarrativeSection; anchored to evidence; cannot be fabricated without evidence.  
**Producers:** NarrativeGenerator.  
**Consumers:** NarrativeSection, candidate-facing report.  
**Persistence:** Stored as part of NarrativeSection.

---

### CoachingPlan

**Purpose:** The complete, session-level study and improvement plan — aggregating LearningObjectives, ranked StudyRecommendations, and prioritisation rationale. The final actionable output delivered to the candidate.  
**Ownership:** CoachingEngine aggregate.  
**Lifecycle:** Assembled after report generation; depends on KnowledgeGaps and Narrative.  
**Producers:** CoachingEngine.  
**Consumers:** ReportBuilder, candidate-facing report.  
**Persistence:** Persisted with SessionHistory.

---

### CoachingAction

**Purpose:** A single, prioritised task within the CoachingPlan (e.g. "Study time complexity analysis — 2 hours — link to resource"). The leaf node of the CoachingPlan tree.  
**Ownership:** CoachingPlan aggregate.  
**Lifecycle:** Generated per KnowledgeGap; ranked globally across all gaps.  
**Producers:** CoachingEngine.  
**Consumers:** CoachingPlan.  
**Persistence:** Stored with CoachingPlan.

---

### LearningProgress

**Purpose:** A cross-session aggregate view of a candidate's dimensional score trends. Shows per-dimension improvement, stagnation, or regression over N sessions.  
**Ownership:** ProgressTracker (services/).  
**Lifecycle:** Computed read-only from SessionHistory; never modifies session records.  
**Producers:** ProgressTracker (reads multiple SessionHistory records).  
**Consumers:** Progress Tracking UI, candidate-facing progress report.  
**Persistence:** Derived from SessionHistory; not independently stored.

---

### LanguageProfile

**Purpose:** The complete language configuration for a session — which programming language(s) are active, the interview mode (single/mixed), and any language-specific evaluation policy overrides.  
**Ownership:** Session configuration / LanguageConfig domain object.  
**Lifecycle:** Set at session start; immutable for session duration; persisted with SessionHistory.  
**Producers:** Session configuration UI / CLI.  
**Consumers:** CodingQuestionPipeline, CodingExecutor, EvaluationEngine, NarrativeGenerator.  
**Persistence:** Persisted with SessionHistory.

---

### ProgrammingLanguage

**Purpose:** A first-class abstract domain concept. Python, JavaScript, TypeScript, Go, Java, Rust, C# are concrete implementations registered in the `LanguageRegistry`. No domain logic references a concrete language by name — all domain logic operates against the abstract concept. **K2 amendment: abstract, not an enum of fixed values.**  
**Ownership:** Language Independence Layer (domain/language/).  
**Lifecycle:** Static registry; extended only when a new language is officially supported (requires LanguageRegistry entry + LanguagePolicy + LanguageExecutor).  
**Producers:** Domain design.  
**Consumers:** LanguageProfile, CodingExecutor dispatch, question generation, evaluation rubric.  
**Persistence:** Code artifact (registry).

---

### CodingCapability

**Purpose:** A language-specific capability assessment — what constructs, idioms, and patterns the candidate demonstrates in a given language during a session.  
**Ownership:** CandidateProfile aggregate (language dimension).  
**Lifecycle:** Derived from coding question evaluation signals; per-language; updated per coding question answered.  
**Producers:** Coding evaluation pipeline.  
**Consumers:** NarrativeGenerator, ProfileFeature extraction.  
**Persistence:** Derived; candidate for V1.3 cross-session accumulation.

---

### LanguagePolicy

**Purpose:** The rules governing how a ProgrammingLanguage is evaluated — what idioms are acceptable, what type errors are flagged, what standard library imports are allowed.  
**Ownership:** Language Independence Layer.  
**Lifecycle:** Static per language version; reviewed at calibration.  
**Producers:** Domain design.  
**Consumers:** EvaluationEngine (language-specific scoring), CodingExecutor.  
**Persistence:** Configuration artifact in evaluation.py or language registry.

---

### SessionHistory

**Purpose:** The durable, complete record of a completed interview session — transcript, answers, evaluation results, NarrativeSections, CoachingPlan, LanguageProfile, metadata.  
**Ownership:** SessionHistory aggregate (persistence layer).  
**Lifecycle:** Written once on session completion; read-only thereafter; forward-compatible schema.  
**Producers:** Session completion pipeline.  
**Consumers:** Replay UI, ProgressTracker, CalibrationProfile (in aggregate).  
**Persistence:** SQLite (durable, local deployment).

---

### Replay

**Purpose:** A read-only, structured rendering of a SessionHistory — question by question, answer by answer, score by score, with coaching notes. No re-submission, no LLM calls.  
**Ownership:** ReplayEngine (app/replay/).  
**Lifecycle:** On-demand; reconstructed from SessionHistory; stateless rendering.  
**Producers:** ReplayEngine.  
**Consumers:** Candidate-facing Replay UI.  
**Persistence:** None (derived from SessionHistory).

---

### Freshness

**Purpose:** The temporal validity attribute of an Observation. An Observation's freshness decays over time (within or across sessions); once it expires, it no longer contributes to ProfileFeature computation.  
**Ownership:** ObservationStore (applies decay logic).  
**Lifecycle:** Assigned at Observation creation; decremented per configured decay function; expires at TTL.  
**Producers:** ObservationStore decay function.  
**Consumers:** FeatureExtractor (filters expired Observations).  
**Persistence:** Stored as a field on Observation.

---

### CalibrationProfile

**Purpose:** The expected score distribution for a given (role, seniority, difficulty, language) configuration — used to validate that scoring constants remain correctly calibrated.  
**Ownership:** Calibration Framework.  
**Lifecycle:** Computed from synthetic session corpus; updated when evaluation constants change; CI gate compares live distribution to CalibrationProfile.  
**Producers:** Calibration test suite.  
**Consumers:** CI gate, evaluation constant change review.  
**Persistence:** Version-controlled artifact in calibration test suite.

### Aggregate Relationships

- `CandidateProfile` is the central profile aggregate. It does not own SessionHistory.
- `SessionHistory` is the persistence aggregate. It receives a snapshot of `CandidateProfile`, `Narrative`, and `CoachingPlan` at session close.
- `CoachingPlan` is owned by the CoachingEngine but stored inside `SessionHistory`.
- `Narrative` is owned by the NarrativeGenerator but stored inside `SessionHistory`.
- No aggregate crosses another's boundary at runtime. Data flows through events and snapshots.

## SECTION E — Language Independence

### Why Language Is a First-Class Domain Concept

In V1.0/V1.1 language was an execution flag. The fundamental problem: language affects question generation, execution runtime, type error semantics, idiomatic evaluation, and — most importantly — should NOT affect evidence signal semantics or evaluation dimension weights.

Making `ProgrammingLanguage` a first-class domain concept enforces this separation permanently.

### Frozen Language Concepts

| Concept | Definition |
|---|---|
| `ProgrammingLanguage` | A typed domain value: `PYTHON`, `JAVASCRIPT`, `TYPESCRIPT`. Not a string. The canonical registry of supported languages. |
| `LanguageProfile` | The runtime language configuration for a session: which language(s) are active, what mode (single/mixed), language version. |
| `LanguageCapability` | A candidate's demonstrated ability in a specific language, derived from coding evaluation signals for that language. |
| `QuestionLanguage` | The language in which a coding question's starter code, constraints, and examples are authored. |
| `ExecutionLanguage` | The language runtime used to execute candidate code. Maps 1:1 to `QuestionLanguage` within a session. |
| `EvaluationLanguage` | The language lens applied during scoring. Language-specific idioms are recognised but do not alter dimension weights or rubric structure. |

### How Python and JavaScript Fit

| Concern | Python | JavaScript / TypeScript |
|---|---|---|
| QuestionLanguage | Python stubs, Python examples | JS/TS stubs, JS/TS examples |
| ExecutionLanguage | AST-guarded Python sandbox | Node.js runtime |
| EvaluationLanguage | Python idiom recognition | JS async/await, TS type error recognition |
| Dimension weights | Unchanged | Unchanged |
| Evidence signal semantics | Language-agnostic | Language-agnostic |

### How Future Languages Fit (Java, Go, Rust, …)

No domain redesign is required. Adding a new language means:
1. Register a new `ProgrammingLanguage` value in the domain registry.
2. Author a `LanguagePolicy` for that language (allowed imports, type semantics).
3. Implement an execution adapter for that runtime.
4. Author question stubs and hidden tests in that language.

All evaluation dimensions, evidence signal semantics, ProfileFeatures, and Narrative structures remain unchanged. The Language Independence Layer guarantees this by design.

- Single file only. No multi-file projects.
- Standard library imports only. No third-party packages.
- No framework questions (React, Django, Express, etc.).
- No database integration questions (SQL handled by SQL engine separately).
- No infrastructure, DevOps, or distributed systems questions.

## SECTION G — Domain Invariants

The following invariants are permanent. No implementation decision in V1.2 or later may violate them without a formal ADR.

| # | Invariant |
|---|---|
| I-01 | **Observations are immutable.** An Observation, once created, is never modified. It can only expire. |
| I-02 | **ProfileFeatures are derived.** A ProfileFeature is never authored directly. It is always the output of a FeatureExtractor reading the ObservationStore. |
| I-03 | **Narratives never generate evidence.** The NarrativeGenerator reads evidence; it does not create or modify Observations or ProfileFeatures. |
| I-04 | **Coaching consumes ProfileFeatures and KnowledgeGaps only.** The CoachingEngine never reads raw EvaluationResults directly. It consumes derived concepts only. |
| I-05 | **Language never changes the semantics of EvidenceSignals.** Signal names, types, and meaning are language-agnostic. A "low reasoning depth" signal means the same thing in Python and JavaScript. |
| I-06 | **EvaluationDimension weights are language-independent.** Correctness, Communication, Reasoning, and Problem Decomposition are weighted identically regardless of programming language. |
| I-07 | **ProfileFeatures own longitudinal knowledge.** If cross-session profiles are introduced (V1.3+), ProfileFeatures are the unit of cross-session accumulation, not raw Observations. |
| I-08 | **Observations own temporal facts.** Observations are the only objects that carry `timestamp` and `freshness`. ProfileFeatures do not carry their own timestamps. |
| I-09 | **SessionHistory is written once and never mutated.** No post-completion modification. Forward-compatible schema additions are additive only. |
| I-10 | **CoachingActions map to KnowledgeGaps.** Every CoachingAction must reference a specific KnowledgeGap. Coaching without an identified gap is not permitted. |
| I-11 | **Replay never invokes LLM calls.** ReplayEngine reconstructs from stored state only. |
| I-12 | **Calibration does not modify evaluation constants.** The CalibrationProfile is a validation artifact, not a tuning mechanism. Constant changes require a separate engineering decision. |
| I-13 | **LanguagePolicy never modifies EvaluationDimension weights.** LanguagePolicy may recognise language-specific idioms but cannot alter the rubric. |
| I-14 | **ObservationStore is append-only.** Observations are added; they are never removed manually. Expiry is the only removal mechanism, applied by the decay function. |
| I-15 | **NarrativeInsights must be anchored.** Every NarrativeInsight must reference at least one Observation or ProfileFeature as its evidence anchor. Evidence-free coaching text is a domain violation. |

## SECTION H — Domain Freeze

### Protected Concepts (stable in V1.2; breaking changes require ADR)

- `Observation` — immutability contract and schema
- `ProfileFeature` — taxonomy, schema, versioning
- `EvidenceSignal` — name registry, type definitions, language-agnostic semantics
- `ObservationStore` — append-only contract, decay interface
- `EvaluationDimension` — dimension names, weight interface
- `ProgrammingLanguage` — value registry
- `SessionHistory` — write-once contract, schema versioning policy

### Reserved Concepts (defined in V1.2 but not fully activated)

- `LanguageCapability` — defined and populated in V1.2; cross-session accumulation reserved for V1.3
- `LearningProgress` — cross-session aggregation in V1.2; cohort-level aggregation reserved for V2
- `TenantContext` — placeholder only in V1.2; full multi-tenant semantics reserved for V2

### Future Concepts (V1.3+, explicitly out of V1.2 scope)

| Concept | Target Version | Rationale |
|---|---|---|
| Cross-session ProfileFeature accumulation | V1.3 | Requires persistent CandidateProfile across sessions |
| PeerBenchmark | V2 | Requires cohort data volume |
| OrganisationProfile | V2 | Requires multi-tenant architecture |
| LiveCoachingEvent | V2 | Requires real-time streaming infrastructure |
| CandidateIdentity | V2 | Requires authentication layer |
| GoalTrack | V1.3 | Requires declared learning objective from candidate |

## SECTION J — Roadmap Validation

### Dependency Analysis

```
EPIC-00 (Language Independence Layer)
  └── No upstream dependencies. Can start immediately.

EPIC-02 (Observation Layer)
  └── Requires ProfileFeature schema from EPIC-01 interface contract.
      However: ObservationStore and ObservationType can be designed in parallel
      with EPIC-01 schema definition. Parallel start is safe.

EPIC-01 (Candidate Knowledge Model — ProfileFeature Activation)
  └── Requires EPIC-02 ObservationStore interface to be stable.
      EPIC-01 is partially blocked by EPIC-02 (FeatureExtractor reads ObservationStore).
      Dependency is narrow: only the read interface, not the full implementation.

EPIC-03 (Narrative Generator V2)
  └── Hard dependency on EPIC-01 (ProfileFeatures available) and EPIC-02 (ObservationStore stable).

EPIC-05 (Evidence Freshness)
  └── Hard dependency on EPIC-02 (ObservationStore decay interface).

EPIC-04 (Coaching Engine)
```
