## SECTION A — Domain Model Validation

The following five amendments are adopted into the canonical V1.2 domain model, superseding the corresponding K1 definitions.

---

### A-1: ObservationStore is an Independent Aggregate Root

**Amendment:** ObservationStore is NOT owned by CandidateProfile. It is a standalone Aggregate Root.

| Concern | K1 (superseded) | K2 (canonical) |
|---|---|---|
| Ownership | CandidateProfile contained ObservationStore | ObservationStore is its own Aggregate Root |
| CandidateProfile | Contained ObservationStore + ProfileFeatures | Consumes ProfileFeatures only; no direct ObservationStore access |
| Boundary | Single aggregate | Two aggregates with a one-way dependency |

**Rationale:** CandidateProfile must remain queryable and serialisable without loading the full observation history. Evidence freshness, replay, and calibration all need to access ObservationStore independently of the profile view.

---

### A-2: Canonical EvidenceSignal → Observation Flow

**Amendment:** Observations are not derived directly from EvaluationResults. The canonical flow is:

```
Evaluation
    ↓
EvidenceSignal          (existing — domain/contracts/reasoning/evidence_signal.py)
    ↓
ObservationExtractor    (NEW — transforms EvidenceSignals into typed Observations)
    ↓
Observation             (first-class domain object — immutable, timestamped, typed)
    ↓
ObservationStore        (append-only aggregate)
    ↓
FeatureEngine           (reads ObservationStore → produces ProfileFeatures)
    ↓
ProfileFeature          (derived, versionable)
    ↓
CandidateProfile        (consumes ProfileFeatures only)
```

**Rationale:** The `EvidenceSignal` abstraction (ADR-033, ADR-046) is already frozen and in production. Inserting `ObservationExtractor` as a translation layer preserves all existing signal contracts while introducing the Observation domain concept cleanly.

---

### A-3: ReasonerDecision Is Explicit in Runtime Flow

**Amendment:** `ReasonerDecision` is a named, explicit runtime object between Evaluation and Observation. The canonical runtime is defined in Section E.

**Rationale:** ADR-030 states "Reasoner outputs are advisory only." ADR-035 states "ReasonerDecision is fully structured." The K1 model treated ReasonerDecision as infrastructure; it is a first-class domain event in the runtime flow.

---

### A-4: FeatureExtractor → FeatureEngine

**Amendment:** The `FeatureExtractor` concept (K1) is renamed and elevated to `FeatureEngine`.

| Concern | K1 | K2 |
|---|---|---|
| Name | FeatureExtractor | FeatureEngine |
| Responsibility | Reads ObservationStore → extracts feature data | Orchestrates multiple Updaters; single producer of all ProfileFeatures |
| Extensibility | Single function | Composable: ObservationUpdater, ReplayUpdater, CalibrationUpdater, future LearningUpdater |
| Ownership | Pattern detection layer | Independent domain service |

**FeatureEngine is the ONLY permitted producer of ProfileFeatures.** No other component may write to `CandidateProfile.features`.

Known Updaters for V1.2:
- `ObservationUpdater` — derives features from current ObservationStore
- `ReplayUpdater` — derives features from SessionHistory (cross-session, V1.3+ full activation)
- `CalibrationUpdater` — validates feature values against CalibrationProfile

---

### A-5: ProgrammingLanguage as Abstract Domain Concept

**Amendment:** `ProgrammingLanguage` is an abstract domain concept. Python, JavaScript, TypeScript, Go, Java, Rust, C# are implementations (concrete instances registered in the `LanguageRegistry`).

```
ProgrammingLanguage  (abstract domain concept)
  ├── Python         (concrete)
  ├── JavaScript     (concrete)
  ├── TypeScript     (concrete)
  └── [Go, Java, Rust, C#, ...]  (future — no domain redesign required)
```

**Invariant:** All domain logic operates against `ProgrammingLanguage` (abstract). No domain concept references a concrete language by name.

## SECTION C — Extension Point Audit

### ADR-039: Evidence Freshness — Architectural Reservation

**Status in TDS:** Accepted (Deferred to V1.2)  
**V1.2 status:** ACTIVATED — This is now EPIC-05.  
**Assessment:** The reservation is sound. `EvidenceSource.DERIVED` is defined but never emitted in V1.1. The `FreshnessWeightedEvidenceStore` wrapper pattern suggested in ADR-039 is compatible with the K2 ObservationStore design (wrapper around existing EvidenceStore is an Observation Intelligence Layer concern).  
**Action required:** New ADR needed to define the TTL policy, decay function shape, and clock interface for testability.  
**No frozen asset changes required.**

---

### ADR-048: ProfileFeature Abstraction — V1.2 Extension Point

**Status in TDS:** Accepted (Architecture direction; implementation reserved for V1.2)  
**V1.2 status:** ACTIVATED — This is the core of EPIC-01.  
**Assessment:** The extension point is well-defined. `CandidateProfile.features: dict[str, ProfileFeature]` with `default_factory=dict` is backward-compatible. The constraint that V1.1 code must not import or reference `ProfileFeature` is satisfied.  
**K2 refinement:** The FeatureEngine (K2 A-4) is the implementation of `CandidateProfileEngine`'s new dispatch to `ProfileFeatureUpdater` described here. No conflict.  
**Action required:** New ADR needed to freeze ProfileFeature schema and versioning policy.  
**No frozen asset changes required.**

---

### ADR-050: NarrativeGenerator Consumes ProfileFeatures, Not Detector Outputs

**Status in TDS:** Accepted (Architecture direction; NarrativeGenerator deferred to M2-8)  
**V1.2 status:** BOUNDARY ENFORCED — This ADR governs EPIC-03.  
**Assessment:** The boundary is clean and unambiguous. NarrativeGenerator receives CandidateProfile as its sole profile input. Access to ReasonerDecision is limited to `follow_up_recommendation` and `navigation_recommendation`.  
**Action required:** None. This ADR is fully active and requires no new decisions.  
**No frozen asset changes required.**

---

### ADR-055: Observation Abstraction — Reserved for V1.2

**Status in TDS:** Proposed (Documentation only; no production code in V1.1)  
**V1.2 status:** ACTIVATED — This is the foundation of EPIC-02 (Observation Intelligence Layer).  
**Assessment:** The proposed abstraction in ADR-055 is consistent with K2 canonical flow. The `ObservationExtractor` (K2 A-2) is the implementation of the "ObservationExtractor" concept sketched in ADR-055.  
**K2 refinement:** ADR-055 shows `Observation → EvidenceSignal → PatternMatch`. K2 corrects this to `EvidenceSignal → ObservationExtractor → Observation → ObservationStore`. EvidenceSignals pre-exist; Observations are derived from them, not the reverse.  
**Action required:** New ADR needed to formalise the ObservationExtractor role and Observation schema (supersedes the sketch in ADR-055).  
**No frozen V1.1 asset changes required.**

### ADR-066: Behavioral Observation Model — V1.2 Extension Contract

**Status in TDS:** Accepted (Architecture direction; implementation V1.2)

V1.2 migration: stat objects become `Observation` subclasses by adding `description: str`. `ObservationType.LEADERSHIP`, `COLLABORATION`, `ADAPTABILITY` are three of the registered types.  
**No conflict with K2 domain model.**  
**Action required:** None. This ADR governs the V1.2 migration contract for the three behavioral detectors already designed.  
**No frozen V1.1 asset changes required.**

## SECTION G — Observation Intelligence Layer

### Renamed from "Observation Layer"

The `Observation Intelligence Layer` is the subsystem responsible for the full lifecycle of Observations within the platform.

### Responsibilities

| Responsibility | Owner |
|---|---|
| **Creation** | ObservationExtractor — transforms EvidenceSignals into typed Observations |
| **Normalisation** | ObservationExtractor — ensures consistent schema, types, and field population |
| **Deduplication** | ObservationStore — rejects duplicate Observations (same type + source + question_index) |
| **Temporal ordering** | ObservationStore — maintains Observations in question_index order |
| **Freshness / Decay** | ObservationStore — applies decay function; expires entries beyond TTL |
| **Replay support** | ObservationStore read interface — provides ordered snapshot for ReplayUpdater |
| **Feature extraction input** | ObservationStore read interface — consumed by FeatureEngine |

### Explicit Boundary

The Observation Intelligence Layer NEVER:
- Generates Narratives
- Generates CoachingActions
- Writes to CandidateProfile
- Invokes LLM calls
- Reads SessionHistory

All downstream consumers (FeatureEngine, CalibrationProfile, ReplayUpdater) read from ObservationStore; they do not write to it.

## SECTION H — ProfileFeature Architecture Audit

### Validated Properties

| Property | Status | Notes |
|---|---|---|
| **Derived** | Confirmed | FeatureEngine is sole producer; no direct authoring |
| **Versionable** | Confirmed | Schema must be frozen and version-controlled (ADR-018) |
| **Immutable** | Confirmed | Once computed by FeatureEngine for a cycle, a ProfileFeature value is frozen |
| **Longitudinal** | Partial in V1.2 | Session-scoped in V1.2; cross-session accumulation reserved for V1.3 via ReplayUpdater |
| **Language-independent** | Confirmed | ProfileFeature taxonomy does not reference any ProgrammingLanguage |
| **Candidate-centric** | Confirmed | Describes the candidate, not the question or the session |

### Future Versioning Strategy (conceptual only)

When a ProfileFeature schema changes:
1. A new version is registered alongside the old (e.g. `ConfidenceCalibration_v1`, `ConfidenceCalibration_v2`).
2. SessionHistory stores the schema version used at generation time.
3. The Replay UI must handle all stored versions.
4. FeatureEngine emits the current version; legacy versions are read-only.
5. A deprecation ADR is required before any version is removed.
