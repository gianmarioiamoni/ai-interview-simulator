## PAT-01 — Engine Five-Artifact Pattern

### Description

When implementing a domain engine (a component that orchestrates domain transformations), the implementation is always structured into exactly five artifacts: **Schema, Protocol, Extractor, Store, and Engine**.

### Purpose

Decompose an engine into independently testable, independently freezable, single-responsibility components. Prevent the engine from becoming a God Object.

### Motivation

During V1.2 architecture design, the `FeatureEngine` (ADR-020) and `ObservationStore` (ADR-017) revealed a recurring decomposition challenge: engines that orchestrate multiple transformations naturally accumulate responsibilities. Without an explicit decomposition rule, they grow into God Objects that are impossible to audit and difficult to extend.

The five-artifact pattern was observed across EPIC-01 (ProfileFeature/FeatureEngine), EPIC-02 (ObservationStore/ObservationExtractor), and EPIC-04 (CoachingPlan/CoachingEngine). Each engine instance followed the same structural decomposition.

### Responsibilities of the Five Artifacts

| Artifact | Responsibility | Immutability Rule | Single-Writer Rule |
|---|---|---|---|
| **Schema** | Defines the domain contract (Pydantic, `extra=forbid`, `schema_version`) | Frozen before any other artifact begins implementation | N/A — pure definition |
| **Protocol** | Defines the extensibility contract for consumers/updaters (ABC or Protocol class) | Frozen after Schema | N/A — pure interface |
| **Extractor** | Transforms input signals into the domain concept; sole writer of the Store | Implemented after Protocol is frozen | Sole writer. No other component may write to its Store. |
| **Store** | Append-only, deduplicated, immutable aggregate; enforces temporal ordering | Implemented after Schema is frozen; API frozen before Extractor is wired | Single writer: the Extractor. Readers are unrestricted. |
| **Engine** | Orchestrates Updaters/Extractors; sole producer of the output concept | Implemented last; replaces all ad-hoc dispatch calls | Sole producer of the output concept. |

### Benefits

- Each artifact can be frozen and certified independently.
- The Extractor and Engine are independently unit-testable without the full pipeline.
- The Store can be verified as append-only in isolation.
- Adding new output dimensions (new Protocol implementations) requires zero changes to the Engine's public interface.
- Architecture audits have a predictable checklist: five artifacts, five ownership verifications.

### When to Use

Use PAT-01 when:
- A component must transform a stream of input signals into a persistent domain model.
- Multiple transformation strategies (updaters, extractors) must be composable.
- The output concept must be independently snapshotable (CandidateProfileSnapshot, KnowledgeSnapshot).
- Downstream consumers must not depend on implementation details of the transformation.

### When NOT to Use

Do NOT use PAT-01 when:
- The component performs a single, non-composable transformation (use a simple service function instead).
- There is only one transformation strategy and no extension point is anticipated.
- The domain concept is a pure value object with no accumulation semantics (use a factory function).
- The lifetime of the component is request-scoped with no persistence requirement.

## PAT-02 — Runtime First, Orchestration Second

### Description

When implementing a domain layer with both a runtime contract (what the component does in a live session) and an orchestration contract (how the component is composed and wired), always freeze and implement the **runtime contract first**. Orchestration is implemented only after the runtime contract is verified.

### Purpose

Prevent orchestration complexity from influencing the design of runtime contracts. Runtime contracts are the domain's public truth; orchestration is an implementation detail.

### Motivation

During V1.2 construction planning, EPIC-01 (FeatureEngine) and EPIC-02 (ObservationStore) revealed a sequencing tension: it was tempting to design the `FeatureEngine` orchestration (how updaters are composed and dispatched) before freezing the `ProfileFeature` schema and `FeatureUpdater` protocol (what the engine produces and how it is extended).

Designing orchestration before runtime contracts produces engines whose API reflects the orchestration strategy rather than the domain model. This made the engine harder to extend and forced API changes when the orchestration strategy changed.

The pattern was formalised: freeze the Schema and Protocol (runtime artifacts) before designing the Engine (orchestration artifact). This is an instance of the more general Manifest principle "Contracts before Code."

### Responsibilities

| Stage | What Is Done | What Is NOT Done |
|---|---|---|
| **Runtime First** | Freeze Schema (`extra=forbid`, `schema_version`). Freeze Protocol (ABC/Protocol interface). Write unit tests for the Schema and Protocol in isolation. | No orchestration code. No Engine. No pipeline wiring. |
| **Orchestration Second** | Implement the Engine after runtime contracts are frozen and tested. Wire the pipeline. Register components. | No changes to frozen Schema or Protocol. |

### Benefits

- Runtime contracts never reflect orchestration convenience.
- The Engine can be replaced (or have its orchestration strategy changed) without touching the Schema or Protocol.
- Downstream consumers can be written against the frozen Schema before the Engine exists.
- Architecture reviews can separately validate runtime correctness and orchestration correctness.

### When to Use

Use PAT-02 when:
- A component has both a per-cycle runtime contract and a session-level orchestration contract.
- Multiple orchestration strategies are anticipated (e.g., sequential vs. parallel updater dispatch).
- Downstream consumers will depend on the runtime contract but not on the orchestration.
- The Schema is intended to be frozen before implementation begins (standard V1.2 workflow).

### When NOT to Use

Do NOT use PAT-02 when:
- The component has no distinct runtime/orchestration boundary (single-step transformations).
- The orchestration is the only reason the component exists (pure adapters, routers).
- The Schema is trivially derivable from the orchestration (configuration-only components).

### Relationship with Existing ADRs

| ADR | Relationship |
|---|---|
| ADR-018 | `ProfileFeature` Schema frozen (Runtime) before `FeatureEngine` orchestration designed |
| ADR-020 | `FeatureEngine` implements the Orchestration layer over the frozen `FeatureUpdater` Protocol |
| ADR-022 | `SessionHistory` schema frozen (Runtime) before `KnowledgeSnapshot` strategy designed (Orchestration) |
| ADR-032 | `CandidateProfileSnapshot` strategy (Orchestration) designed after `ProfileFeature` schema is frozen |
| Manifest §3 | "Contracts before Code" — PAT-02 is the domain-engine–specific application of this principle |

## PAT-03 — Construction Parallelism Review (CPR)

### Description

Before any construction phase begins, perform a **Construction Parallelism Review**: a structured analysis of which implementation tracks can proceed in parallel, which tracks are serialised by hard dependencies, and what the critical path is. The result is a frozen parallel execution plan.

### Purpose

Maximise construction throughput while preserving dependency integrity. Prevent teams from waiting unnecessarily on dependencies that do not yet exist, or from building components that depend on unstable upstream contracts.

### Motivation

V1.2 has eight Epics with non-trivial cross-dependencies. Without a parallelism review, the natural tendency is to execute Epics sequentially (EPIC-00 complete → EPIC-01 start) even when phases within Epics could proceed in parallel.

The V1.2 Implementation Roadmap (§J) conducted a CPR that identified:
- **Immediately parallelisable:** EPIC-00 (Language Independence) + EPIC-02 M1-M3 (ObservationStore design, no pipeline integration) + ADR authoring (ADR-024, ADR-029)
- **Critical path serial dependency:** EPIC-00 → EPIC-01 → EPIC-02 (full) → EPIC-03 → EPIC-04 → EPIC-06
- **Quick-win parallel tracks:** EPIC-05 (Evidence Freshness) after EPIC-02 M3; EPIC-07 (Enterprise Extensibility) after persistence schema stable

Without CPR, EPIC-02 M1-M3 would have waited until EPIC-00 was complete, adding unnecessary delay to the critical path.

### Responsibilities

A CPR produces three outputs:

| Output | Description |
|---|---|
| **Dependency Graph** | Explicit statement of which component requires which upstream component before it can begin. Distinguished from "would benefit from" vs "is blocked by." |
| **Parallel Tracks** | List of implementation tracks that have no hard dependency on each other within their current phase. Each track must be independently mergeable (no shared mutable contracts). |
| **Critical Path** | The serialised sequence of components that determines the minimum construction duration. No parallel track accelerates the critical path unless it is on it. |

### Benefits

- Eliminates artificial serialisation caused by conservative dependency assumptions.
- Produces an explicit, auditable execution plan that can be reviewed against the Implementation Baseline.
- Identifies quick wins (low-dependency, high-value tracks) that can be started immediately.
- Makes the critical path visible so risk mitigation can focus on it.
- Prevents teams from starting parallel tracks that share an unfrozen contract (which creates merge risk).

### When to Use

Use PAT-03 at the start of every Construction Phase:
- When more than three Epics or milestones are in scope.
- When Epics have cross-Epic dependencies (one Epic's output is another's input).
- When the team has capacity to execute multiple tracks simultaneously.
- When a construction phase has a target delivery date (critical path awareness is essential).

### When NOT to Use

Do NOT use PAT-03 when:
- Only one Epic or milestone is in scope (no parallelism to review).
- All Epics are strictly sequential with no opportunity for parallelism.
- The team has capacity for only one track at a time (CPR is still valid but produces no actionable parallelism).

### Relationship with Existing ADRs

| Document | Relationship |
|---|---|
| V1.2-IMPLEMENTATION-BASELINE §B2 | The "Parallel Tracks (within Phase)" table is the CPR output for V1.2 |
| V1.2-IMPLEMENTATION-ROADMAP §J | The "Recommended Sequence" and "Parallelizable Work" tables are the V1.2 CPR execution plan |
| Playbook §C | Milestone Workflow defines the serialised within-milestone sequence; CPR governs the cross-milestone parallelism |
| Manifest §9 | "Composition over inheritance" — CPR relies on components being composable and independently deployable |

### Examples from V1.2

**V1.2 Construction Parallelism Review result (Phase 1):**

```
Immediately parallelisable (no shared dependency):
├── EPIC-00: Language Independence Layer (M) — no upstream dependency
├── EPIC-02 M1-M3: ObservationStore + ObservationType — no consumer dependencies yet
└── ADR-024 + ADR-029 authoring — non-blocking Support ADRs

Serial dependency (cannot parallelise):
EPIC-00 → EPIC-01 (ProfileFeature needs LanguageConfig stable)
EPIC-01 → EPIC-02 full (ObservationUpdater needs ProfileFeature schema frozen)
EPIC-02 → EPIC-03 (NarrativeGenerator needs ObservationStore stable)
EPIC-03 → EPIC-04 (CoachingEngine needs Narrative stable)
```

**Dependency classification used in CPR:**

| Classification | Meaning | Example |
|---|---|---|
| **Hard block** | Component B cannot be started until Component A is frozen | EPIC-01 M3 needs ProfileFeature schema (E01-M1) frozen |
| **Interface stub** | Component B can be started against a stub of A's interface, not the frozen API | EPIC-01 M3 (ObservationUpdater) can start against ObservationStore interface stub |
| **Independent** | No shared contracts; can proceed in parallel | EPIC-00 and EPIC-02 M1-M3 |

## PAT-04 — Temporary Construction Placeholder (TCP)

### Description

When a V2 or future-milestone capability is architecturally desirable in V1.2 schemas but has no functional behaviour in V1.2, introduce an explicit **nullable, no-op placeholder** in the schema at construction time. The placeholder is:
- Additive only (never replaces an existing field).
- Nullable with a documented default of `null`.
- Assigned a governing ADR before implementation.
- Identified as a placeholder in the schema (via field description or naming convention).
- Removed from the V1.2 functional path (no V1.2 feature depends on it being non-null).

### Purpose

Reserve schema migration paths for future versions at zero implementation cost in the current version, while preventing future breaking schema changes.

### Motivation

During V1.2 architecture design, ADR-016A (`CandidateIdentity`) and ADR-022 (`SessionHistory`) identified that V2 will require multi-tenant support. Adding `tenant_id` in V2 would require a schema migration across all V1.x session records. Adding it as a nullable no-op in V1.2 costs nothing and eliminates the V2 migration complexity.

Similarly, ADR-029 (TenantContext Placeholder) was identified as a Support ADR: its entire purpose is to specify how the TCP is introduced without requiring any functional V1.2 code to change.

The pattern emerged from the broader Manifest principle "Progressive Evolution" — each version builds on frozen extension points of the previous version. TCP is the implementation-level mechanism for that principle.

### Responsibilities

| Stage | Responsibility |
|---|---|
| **Architecture design** | Identify which future-milestone capabilities will require schema fields that do not yet exist. Document in the relevant ADR (e.g., ADR-016A, ADR-029). |
| **Schema authoring** | Add the placeholder field as nullable with a default value. Mark as reserved in field description. Apply `extra=forbid` to prevent undocumented use. |
| **Implementation** | EPIC-07 (or equivalent) implements the placeholder: schema field added, storage column added (nullable, default null), V2 migration path documented. |
| **Verification** | All V1.2 sessions run identically with the placeholder field null. No V1.2 feature depends on the placeholder being populated. Architecture audit confirms no-op status. |
| **Future activation** | The activating version authors a new ADR that references the reserved field, populates it, and removes the "placeholder" designation. No schema migration required. |

### Benefits

- Zero cost at the current version: nullable fields have no runtime impact.
- Zero migration cost at the activating version: field already exists in all records.
- Eliminates the risk of a future version requiring a breaking schema change.
- Makes architectural intent explicit (future reviewers understand why the field exists).
- Preserves the Manifest invariant "Backward compatibility": additive evolution is possible precisely because placeholders are reserved in advance.

### When to Use

Use PAT-04 when:
- A V2 (or future milestone) capability requires a schema field that does not exist today.
- The field is either optional or can default to `null` / an empty equivalent.
- The field's absence does not impair any V1.x functionality.
- A governing ADR can be authored for the placeholder before the current version closes.

### When NOT to Use

Do NOT use PAT-04 when:
- The future capability requires a non-nullable field (this is a breaking change that cannot be reserved as a TCP).
- The future capability's schema shape is not yet known (unknown fields are not reserved — they are designed when the capability is designed).
- The placeholder would require production code to behave differently (any conditional logic on a TCP field is a design violation).
- The field could be misinterpreted as a production feature by consumers (if naming ambiguity exists, resolve it before introducing the TCP).

### Relationship with Existing ADRs

| ADR | Relationship |
|---|---|
| ADR-016A | `CandidateIdentity` schema includes `tenant_id: Optional[str] = None` as a TCP for V2 multi-tenant |
| ADR-022 | `SessionHistory` schema includes `tenant_id` column as a TCP |
| ADR-029 | The governing ADR for the V1.2 TCP activation: specifies exactly how `TenantContext` is inserted as a no-op nullable field |
| Manifest §11 | "Progressive Evolution" — TCP is the implementation mechanism for this principle |
| Manifest §12 | "Backward Compatibility" — TCP enables additive evolution without breaking frozen V1.x contracts |
| Manifest §3 | "Immutability by default" — TCP fields are immutable null values in V1.2; they do not mutate anything |
| Playbook §L | "Extension Points" lesson — "The cost of defining a reservation is negligible. The cost of missing one is a breaking change." |

### Examples from V1.2


## PAT-05 — Builder-only Construction

### Description

Domain objects with complex construction logic are created exclusively through dedicated builder components. No application service, node, or orchestrator constructs the object directly. The builder is the sole construction path.

### Purpose

Prevent construction logic from leaking into orchestrators, nodes, or application services. A builder is the Single Creation Path for its target object.

### Motivation

During V1.2 Construction, repeated audit findings showed that domain object creation was duplicated across orchestration nodes and services when no explicit builder pattern was enforced. Duplicated construction logic produces inconsistent objects and multiple implicit creation paths that cannot be audited individually.

The pattern was extracted from observed practice across EPIC-01 (`FeatureEngine` building `ProfileFeature`) and EPIC-04 (`CoachingEngine` building `CoachingPlan`): in all cases where a dedicated builder existed, the domain object was consistent and auditable. Where no builder was designated, inconsistencies emerged.

### Responsibilities

| Stage | Responsibility |
|---|---|
| **Builder** | Sole owner of object construction logic. Validates all invariants. Returns a frozen domain object. |
| **Caller** | Provides raw input materials. Never assembles the target object itself. |
| **Domain object** | `extra=forbid`; immutable after construction. Accepts no partial states. |

### Benefits

- Single Construction Path: every instance of the domain object is created identically.
- Construction logic is independently testable without the full pipeline.
- Architecture audits have a predictable check: one builder per constructed domain object.
- No hidden construction paths in orchestrators or test utilities.

### When to Use

Use PAT-05 when:
- A domain object requires invariant validation during construction.
- Multiple callers would otherwise each replicate partial construction logic.
- The domain object is immutable after creation.

### When NOT to Use

Do NOT use PAT-05 when:
- The domain object is a simple value object constructable from a single expression.
- No validation is required and the constructor is sufficient.

### Relationship with Existing ADRs

| ADR | Relationship |
|---|---|
| ADR-020 | `FeatureEngine` is the builder of `ProfileFeature` aggregates |
| ADR-025 | `CoachingEngine` is the builder of `CoachingPlan` |
| Manifest §4 | "Small independent components; Single Responsibility" |
| Manifest §Single Writer Principle | Builder-only construction is the construction-time instance of the Single Writer rule |

## PAT-06 — Single Runtime Orchestrator

### Description

Exactly one runtime component owns workflow execution, branching, retries, recovery, and lifecycle management. In V1.2, this owner is **LangGraph**. Application services are orchestrated by LangGraph nodes. No application service may orchestrate another application service at the workflow level.

### Purpose

Prevent the emergence of a second runtime orchestrator. A system with two orchestrators has an ambiguous execution model: which orchestrator governs retries? Which owns the lifecycle? This ambiguity produces hidden coupling, unpredictable failure modes, and untestable workflows.

### Motivation

During the V1.2 Construction Architecture Review (CAR), `InterviewPipeline` was identified as a second runtime orchestrator: it coordinated multiple application services directly, duplicating workflow responsibilities already owned by LangGraph nodes. This created two parallel execution models in the same system.

`InterviewPipeline` was intentionally removed. All workflow coordination was consolidated into LangGraph nodes. This was the triggering event that formalised PAT-06.

### LangGraph Ownership

LangGraph is the Single Runtime Orchestrator. It owns:

- Workflow sequence and branching
- Conditional edges and state transitions
- Retries and failure recovery
- Session lifecycle management

Application services own only their business capability (evaluation, humanization, reasoning). They are stateless with respect to workflow.

### Responsibilities

| Owner | Owns | Does NOT Own |
|---|---|---|
| **LangGraph** | Workflow, branching, retries, recovery, lifecycle | Business logic, domain rules |
| **Application services** | Business capability (single responsibility) | Workflow coordination, service-to-service orchestration |

### Benefits

- Single execution model: the workflow is readable from the LangGraph graph definition alone.
- All retry and recovery logic is centralised in one place.
- Application services remain independently testable without a running graph.
- Architecture audits have a clear check: does any service call another service at the workflow level?

### When to Use

PAT-06 is always active. It is an invariant, not an optional pattern. Any component that begins coordinating other services at the workflow level violates PAT-06 and must be removed or restructured.

### Identifying a Violation

A component violates PAT-06 when it:
- Calls two or more application services in sequence to produce a workflow outcome.
- Contains branching logic based on workflow state (not domain state).
- Implements retry or recovery for a workflow step.
- Manages session lifecycle transitions.

### Relationship with Existing ADRs and Documents

| Reference | Relationship |
|---|---|
| TDS §21 | LangGraph as the sole runtime — EvaluationSignalWriter flow confirms single-graph execution |
| Manifest §Runtime Ownership (new) | PAT-06 is the pattern implementation of the Runtime Ownership invariant |
| Manifest §Single Writer Principle | PAT-06 extends Single Writer from data ownership to runtime ownership |
| PAT-03 (CPR) | Construction parallelism reviews must verify PAT-06 compliance across all parallel tracks |

### Examples from V1.2

**InterviewPipeline removal (Construction Architecture Review finding):**

```
Finding:    InterviewPipeline coordinated EvaluationService + HumanizerService
            at the workflow level — second orchestrator.
Resolution: InterviewPipeline removed.
            All coordination moved into LangGraph nodes (question_node, reasoner_node).
Result:     Single Runtime Orchestrator invariant restored.
```

**Compliant service design:**

```
LangGraph node: reasoner_node
  → calls ReasonerService.reason()       ← business capability only
  → calls EvaluationSignalWriter.write() ← business capability only
  → updates InterviewState               ← owned by LangGraph
```

## Construction Architecture Review (CAR)

### Purpose

The Construction Architecture Review (CAR) is a mandatory structured review performed after a construction phase and before the pattern extraction and freeze cycle. Its purpose is to identify architectural violations, structural regressions, and emerging patterns that appeared during implementation but were not captured in the architecture design phase.

CAR is the bridge between Construction and Methodology Evolution.

### Position in the Engineering Lifecycle

```
Architecture
    ↓
Construction
    ↓
CAR  ← mandatory gate
    ↓
Remediation Sprint
    ↓
Pattern Extraction
    ↓
Freeze
    ↓
Next Construction Sprint
```

### Inputs

- Completed implementation from the preceding construction phase.
- Accepted ADRs governing the construction phase.
- Implementation Baseline (SECTION B — workflow and quality gates).
- Audit reports from the construction phase.

### Outputs

- CAR Report: structured list of findings, classified by severity (P0–P3).
- Remediation Backlog: ordered list of corrections required before freeze.
- Pattern Candidates: newly observed patterns that recurred ≥ 2 times and qualify for extraction.
- Runtime Ownership Verification: explicit confirmation that PAT-06 is satisfied.

### Acceptance Criteria

A CAR is complete when:
- All P0/P1 findings are resolved or registered with target milestone.
- All Forbidden technical debt is eliminated.
- Runtime ownership is verified: no second orchestrator present.
- Pattern candidates are identified and handed to Pattern Extraction.
- CAR Report is registered in INDEX.

### Typical Findings

- Second runtime orchestrators (PAT-06 violation — InterviewPipeline pattern).
- Duplicated construction paths (PAT-05 violation).
- Services owning workflow state.
- Construction-time coupling introduced between previously independent services.
- Implicit fallbacks without documented fallback policy.

### Relationship with Remediation Sprint

The CAR produces the Remediation Backlog. The Remediation Sprint consumes that backlog and resolves all P0/P1 findings before the freeze gate is opened. No pattern is extracted from a finding that has not been remediated.

### Recommended Frequency

One CAR per construction phase. For V1.2, this is one CAR after all eight Epics complete their implementation tracks. For future versions with staged delivery, one CAR per Epic cluster (group of Epics with shared architectural scope).

## Pattern Extraction

### Definition

Pattern Extraction is the engineering practice of identifying recurring structural practices from implementation, validating that they are genuinely reusable, documenting them, and freezing them as named patterns. Patterns are never invented first. They emerge from implementation.

### Lifecycle

```
Observation (pattern appears once in implementation)
    ↓
Repeated occurrence (pattern appears in ≥ 2 independent contexts)
    ↓
Extraction (pattern is named, scope is defined)
    ↓
Validation (pattern is reviewed against Manifest principles)
    ↓
Documentation (Purpose, Motivation, Responsibilities, Benefits, When to Use)
    ↓
Freeze (pattern registered in V1.2-PATTERN-FREEZE.md and Playbook §N)
    ↓
Reuse (pattern is the default practice for its applicable context)
```

### Rules

- A pattern requires ≥ 2 independent occurrences before it qualifies for extraction.
- A pattern that contradicts a Manifest principle is not extracted — the implementation is corrected instead.
- A pattern must be named before it can be enforced. Unnamed practices cannot be referenced in audit reports.
- Pattern extraction occurs after CAR, not during construction.
- The human architect accepts patterns. AI tooling may propose candidates.

### Relationship with CAR

The CAR identifies pattern candidates as part of its output. Pattern Extraction processes those candidates. The two activities are sequential, not parallel.

## Runtime Ownership

### Definition

Runtime ownership defines which component is the authoritative owner of workflow execution, lifecycle management, and recovery behaviour. Runtime ownership is a first-class architectural invariant, equivalent in authority to data ownership (Single Writer Principle).

### Ownership Assignment (V1.2)

| Owner | Runtime Responsibilities |
|---|---|
| **LangGraph** | Workflow sequence, conditional branching, retries, recovery, session lifecycle |
| **Application services** | Business capability within a single node invocation |

### Invariants

1. LangGraph is the Single Runtime Orchestrator. No other component may claim runtime ownership.
2. Application services are stateless with respect to workflow. They receive input, produce output, and return.
3. No service may call another service to produce a workflow outcome. Service-to-service coordination belongs in LangGraph nodes.
4. Recovery logic (retry, fallback, circuit-breaker at the workflow level) belongs in LangGraph, not in services.

### Enforcement

Runtime ownership violations are P1 findings in any CAR or architecture review. The finding class is: "second runtime orchestrator." Resolution requires removal of the violating component or restructuring so that all workflow coordination returns to LangGraph.

## Methodology Evolution

### How Engineering Knowledge Becomes Methodology

The methodology of this project is not designed in advance. It evolves from engineering experience. New knowledge follows this lifecycle:

```
Observation (something recurring is noticed in implementation)
    ↓
Repeated occurrence (it appears in ≥ 2 independent contexts)
    ↓
Engineering Pattern (it is named, extracted, and documented)
    ↓
Pattern Freeze (it is registered and becomes the default practice)
    ↓
Methodology (it is codified in the Playbook and Manifest)
    ↓
Baseline (it is the expected starting point for the next version)
```

### Principles

- Methodology is discovered, not imposed. No practice is codified before it has proven its value in implementation.
- A pattern that appears only once is an observation, not a rule.
- A pattern that contradicts implementation experience is revised, not defended.
- The Manifest defines invariants. The Playbook defines methodology. Patterns implement both.

### Relationship Between Releases

Each version of the platform produces a methodology update:

| Version | Methodology contribution |
|---|---|
| V1.1 | Manifest + Playbook established. Lessons from V1.1 construction formalised (Playbook §L). |
| V1.2 DOC-M1 | PAT-01 to PAT-04 extracted from V1.2 Construction planning. Pattern Registry established. |
| V1.2 RC-C | PAT-05 to PAT-06 extracted from V1.2 Construction Phase. CAR formalised. Pattern Extraction formalised. Runtime Ownership formalised. Methodology Evolution documented. |
