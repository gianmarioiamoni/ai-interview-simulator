## SECTION A — V1.2 Vision

### Vision Statement

V1.2 transforms the AI Interview Simulator from a single-session practice tool into a persistent, longitudinal coaching platform. Where V1.1 established evaluation depth and conversational realism, V1.2 adds memory, continuity, and a candidate knowledge model that grows richer with every session.

### What V1.2 Adds

- **Candidate Knowledge Model** — `ProfileFeature` activation: the `CandidateProfile` built by the Interview Reasoner in V1.1 is now consumed downstream by the `NarrativeGenerator` and `ReportBuilder`, producing coaching language grounded in observed behavioral patterns.
- **Session Persistence & Replay** — durable session storage (SQLite) with a full transcript replay UI.
- **Progress Tracking** — cross-session dimensional trend charts and improvement metrics.
- **Study Recommendations Engine** — gap-to-resource mapping producing a prioritised study plan per session.
- **Evidence Freshness** — stale observation decay; only recent signals influence the candidate profile.
- **Calibration Framework** — systematic validation that scoring constants remain correctly calibrated across role/seniority/language dimensions.
- **Cost Optimisation Framework** — model routing, token budgeting, retrieval caching.
- **Language Independence Layer (official)** — Python, JavaScript, TypeScript formally encoded as a first-class architecture concern with explicit interview configuration modes.

### What V1.2 Deliberately Does NOT Add

| Excluded | Reason |
|---|---|
| REST API public surface | V2 milestone; requires auth layer + multi-tenant infra |
| Enterprise Analytics dashboard | V2 milestone; requires PostgreSQL + organisation model |
| Multi-modal input (video, screen share) | V2; infrastructure investment exceeds V1.2 scope |
| SaaS subscription/billing | V2 business model concern |
| Peer benchmarking | Requires cohort data volume not available at V1.2 |
| Evaluated follow-up (ADR-010 Model B) | Deferred; model complexity vs. session quality trade-off |
| Framework-level coding questions | Intentionally excluded — see Section E |
| Multi-file coding projects | Intentionally excluded — see Section E |
| New interview types beyond Written/Coding/SQL | Scope freeze |

### Why

V1.2 completes the coaching loop that V1.1 opened. V1.1 built the Reasoner and the `CandidateProfile`; V1.2 makes that model visible and actionable. Every V1.2 feature serves either longitudinal continuity (replay, progress, recommendations) or scoring integrity (calibration, freshness, cost control). Features that require new infrastructure tiers (REST API, enterprise auth, multi-modal pipelines) are deferred to V2.

**Dependencies:** V1.1 multi-language execution sandbox (EPIC-08).  
**Expected Outputs:** `LanguageConfig` domain object; interview session config extended with language mode; all question generation, evaluation, and execution paths language-parametric.

### EPIC-01 — Candidate Knowledge Model (ProfileFeature Activation)

**Purpose:** Connect the `CandidateProfile` and `ProfileFeature` objects built by the Interview Reasoner (V1.1) to the `NarrativeGenerator` and `ReportBuilder`, making behavioral evidence visible in coaching output.  
**Dependencies:** V1.1 Interview Reasoner (EPIC-04 complete), M2-9 deferred items.  
**Expected Outputs:** `NarrativeGenerator` reads `ProfileFeatures`; `ReportBuilder` inserts profile-driven coaching sections; `ProfileFeature` schema frozen and version-controlled.  
**Acceptance Criteria:** Reports contain coaching language referencing at least one detected `ProfileFeature` for sessions where pattern detectors fired; no regression on V1.1 evaluation output.  
**Priority:** P0

### EPIC-02 — Observation Layer

**Purpose:** Formalise how raw evaluation signals are converted to `Observation` objects, stored on the `CandidateProfile`, and consumed by pattern detectors and the narrative pipeline.  
**Dependencies:** EPIC-01 (ProfileFeature schema).  
**Expected Outputs:** `ObservationStore` with typed schema; detector input/output contracts frozen; `Observation` lifecycle (creation → decay → expiry) defined.  
**Acceptance Criteria:** Every pattern detector receives correctly typed `Observation` inputs; `ObservationStore` accumulates correctly across multi-question sessions; stale observations do not influence profile after configured TTL.  
**Priority:** P0

---

### EPIC-03 — Narrative Generator

**Purpose:** Upgrade `NarrativeGenerator` to read `CandidateProfile.profile_features` and produce coaching text that references specific behavioral patterns, not only dimensional scores.  
**Dependencies:** EPIC-01, EPIC-02.  
**Expected Outputs:** `NarrativeGenerator` V2 implementation; prompt templates for profile-aware coaching language; structured output schema for narrative sections.  
**Acceptance Criteria:** Narrative sections contain at least one profile-feature reference when applicable; LLM structured output schema validation passes ≥ 99% of sessions; coaching language is non-generic (references answer-specific signals).  
**Priority:** P0

---

### EPIC-04 — Coaching Engine

**Purpose:** Produce the Study Recommendations Engine — a gap-to-resource mapping that generates a prioritised study plan per session.  
**Dependencies:** V1.1 Knowledge Gap Engine (EPIC-02), EPIC-03 (narrative).  
**Expected Outputs:** `StudyRecommendationEngine`; curated resource library (version-controlled); per-session recommendation report section; ranking by gap severity × IRS impact.  
**Acceptance Criteria:** 100% of sessions with identified gaps produce a non-empty ranked recommendation list; recommendations reference gap-specific resources; zero-gap sessions produce a completion acknowledgement message.  
**Priority:** P1

---

### EPIC-05 — Evidence Freshness

**Purpose:** Implement observation decay so that early-session signals do not permanently distort the candidate profile for long-running or cross-session users.  
**Dependencies:** EPIC-02 (ObservationStore), EPIC-01 (ProfileFeature lifecycle).  
**Expected Outputs:** Decay function (configurable TTL per observation type); `ObservationStore` expiry logic; profile recalculation on each detector cycle after decay.  
**Acceptance Criteria:** Observations beyond configured TTL do not contribute to profile score; freshness configuration is centralised in `settings.py`; no regression on active-session behaviour (TTL > session duration default).  
**Priority:** P1

---

### EPIC-06 — Calibration Framework

**Purpose:** Systematically validate that scoring constants in `infrastructure/config/evaluation.py` remain correctly calibrated as question corpus, language coverage, and candidate profiles evolve.  
**Dependencies:** V1.1 evaluation pipeline, EPIC-00 (language config).  
**Expected Outputs:** Calibration test suite with synthetic session corpus; per-dimension score distribution reports; calibration CI gate (fail if score distribution drifts beyond threshold).  
**Acceptance Criteria:** Calibration suite runs in CI; score distribution for synthetic sessions is within expected range per difficulty/seniority matrix; any change to evaluation constants triggers recalibration run.  
**Priority:** P1

---

### EPIC-07 — Enterprise Extensibility

**Purpose:** Add hook points that allow V2 to introduce multi-tenant organisation accounts, per-org configuration overrides, and analytics partitioning without requiring a full architecture rewrite.  
**Dependencies:** EPIC-05 (freshness), session storage (EPIC-09 V1.2 persistence).  
**Expected Outputs:** `TenantContext` placeholder in session state (nullable, no-op in V1.2); per-session config isolation pattern documented; storage schema includes `tenant_id` column (null for V1.2 personal sessions).  
**Acceptance Criteria:** All V1.2 sessions run identically with `tenant_id = null`; schema migration path to multi-tenant V2 is documented; no V1.2 feature depends on tenant context being populated.  
**Priority:** P2

## SECTION D — Multi-Language Coding Vision

### Language Independence Layer

V1.2 officially introduces the **Language Independence Layer** — an architectural principle stating that no evaluation, generation, or execution component may hardcode assumptions about a specific programming language.

### Supported Languages in V1.2

| Language | Status |
|---|---|
| Python | Supported (V1.0+) |
| JavaScript | Supported (V1.1+) |
| TypeScript | Supported (V1.1+) |

### Interview Configuration Modes

| Mode | Description |
|---|---|
| Python only | All coding questions delivered in Python |
| JavaScript / TypeScript only | All coding questions delivered in JS/TS (candidate selects at session start) |
| Mixed (50% Python + 50% JS/TS) | Session alternates languages; tests both language families |

### Why This Architecture Is Language-Independent

- **Question generation** is language-parametric: the `CodingQuestionPipeline` receives a `LanguageConfig` and produces language-appropriate stubs, constraints, and examples.
- **Execution sandbox** is language-dispatched: Python uses the AST-guarded sandbox; JS/TS uses the Node.js runtime. The `CodingExecutor` selects the correct sandbox from `LanguageConfig.runtime`.
- **Evaluation rubric** is language-agnostic: correctness, communication, reasoning, and problem decomposition apply uniformly. Language-specific idioms are not penalised.
- **Hidden test cases** are authored per language but share the same JSON schema, enabling uniform test result parsing.
- **No new language** requires changes outside the `LanguageConfig` registry and the corresponding execution adapter.

## SECTION E — Coding Scope Freeze

### Supported Question Categories

| Category | Status |
|---|---|
| Function implementation | ✓ Supported |
| Small algorithms | ✓ Supported |
| Debugging | ✓ Supported |
| Code comprehension | ✓ Supported |
| Small refactoring | ✓ Supported |
| Data manipulation | ✓ Supported |
| Basic OOP snippets (single file only) | ✓ Supported |

### Explicitly Excluded

| Category | Reason |
|---|---|
| Frameworks (React, Angular, Vue, Express, NestJS, Django, Flask, FastAPI, Spring, ASP.NET) | Framework questions require runtime environment setup, dependency resolution, and UI interaction — incompatible with the sandboxed single-file execution model. Evaluation validity cannot be guaranteed without the framework present. |
| Database integration | Requires a live database connection; SQL questions are handled by the dedicated SQL engine with in-memory SQLite, not the coding executor. |
| Docker / Kubernetes / Infrastructure | Infrastructure as code is not a coding interview category. System design questions cover architectural reasoning separately. |
| Multi-file projects | The execution sandbox is single-file by design. Multi-file projects introduce import resolution, build tooling, and test framework dependencies that break sandbox isolation. |
| Distributed systems / Large architectures | These are system design questions, not coding questions. A different evaluation rubric applies. |

### Why These Exclusions Are Permanent for V1.2

The coding execution model is intentionally minimal: one file, one function or class, one set of hidden test cases, deterministic output comparison. Expanding scope to frameworks or multi-file projects would require a full container-based execution environment, a dependency management layer, and a fundamentally different evaluation rubric. These belong to a future "Advanced Coding Engine" milestone beyond V1.2.

| Constraint | Value |
|---|---|

| Difficulty levels | Easy / Medium / Hard (aligned to seniority: Junior / Mid / Senior) |

## SECTION I — Risks

### Architectural Risks

| Risk | Mitigation |
|---|---|
| `CandidateProfile` schema drift between V1.1 detectors and V1.2 consumers | Freeze `ProfileFeature` schema at EPIC-01 start; version-control it in `domain/contracts/` |
| Observation decay TTL misconfiguration distorting active sessions | Default TTL > max session duration; integration tests validate TTL boundary behaviour |
| Language Independence Layer introducing dispatch complexity | `LanguageConfig` is a single value object; dispatch is a single switch point in `CodingExecutor` |

### Scope Risks

| Risk | Mitigation |
|---|---|
| EPIC-01 `NarrativeGenerator` scope creep (adding new coaching dimensions) | Limit V1.2 to `ProfileFeature` consumption only; no new detector authoring in V1.2 |
| Study Recommendations resource library becoming stale immediately | Resource library is version-controlled and reviewed at release gate; staleness is a V2 concern |
| Calibration Framework expanding into a full A/B testing framework | Scope strictly to scoring constant validation; no model comparison or experiment tracking in V1.2 |

### Engineering Risks

| Risk | Mitigation |
|---|---|
| Session storage schema migration complexity across V1.1 → V1.2 | Forward-compatible schema design; migration is additive only (no column drops) |
| Evidence Freshness decay function producing non-deterministic test behaviour | Decay is clock-based; tests use a mockable clock interface |
| Cost Optimisation model routing producing lower-quality evaluation output | Cheaper model tier only for low-complexity calls (humanizer framing); evaluation reasoning stays on primary model |

### Product Risks

| Risk | Mitigation |
|---|---|
| Coaching language quality regression when `NarrativeGenerator` reads `ProfileFeatures` | Behavioural validation suite (20 scenarios minimum) before EPIC-03 release gate |
| Progress tracking producing misleading trends when role/seniority changes between sessions | Sessions are segmented by (role, seniority) pair; cross-segment comparisons are explicitly labelled |
