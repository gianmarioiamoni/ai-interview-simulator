## SECTION B1 — Implementation Philosophy

### Frozen Principles

> **Architecture is immutable during implementation.**  
> Architecture is designed first. Implementation follows. Implementation never changes architecture.

1. **Architecture First.** No implementation decision may alter an architectural boundary, aggregate ownership, or extension point without a superseding ADR.
2. **Implementation Follows.** Code, tests, and contracts are downstream of architecture. They express the architecture; they do not redefine it.
3. **ADR Authority.** Any implementation that deviates from an accepted ADR is a defect, not an architectural evolution.
4. **Contract Stability.** Frozen contracts (`extra=forbid`, `schema_version`) are implementation invariants. No implementation may loosen them.
5. **Single Writer.** Every aggregate has one writer. Implementation must not introduce second writers under any circumstance.

## SECTION B2 — Implementation Order

### Frozen Roadmap

```
EPIC-00   Language Independence Layer
  ↓
EPIC-01   Candidate Knowledge Model (ProfileFeature Activation)
  ↓
EPIC-02   Observation Layer
  ↓
EPIC-03   Narrative Generator V2
  ↓
EPIC-04   Coaching Engine
  ↓
EPIC-05   Evidence Freshness
  ↓
EPIC-06   Calibration Framework
  ↓
EPIC-07   Enterprise Extensibility
```

### Rationale

| Epic | Why This Position |
|---|---|
| **EPIC-00** | Language Independence Layer has no upstream dependencies. It is an isolation layer that must be in place before any language-parametric component is implemented. Starting here eliminates all future hardcoded language assumptions. |
| **EPIC-01** | ProfileFeature schema and FeatureEngine architecture must be frozen before EPIC-02 (ObservationStore) can be implemented against a stable consumer interface. EPIC-01 interface must be stable first. |
| **EPIC-02** | ObservationStore + ObservationExtractor are the sole writers to the ObservationStore aggregate. Must be implemented after EPIC-01 schema is stable so that FeatureEngine can read from it correctly. |
| **EPIC-03** | NarrativeGenerator V2 depends on stable ProfileFeatures (EPIC-01) and ObservationStore (EPIC-02). It is the first consumer of the full knowledge pipeline. |
| **EPIC-04** | CoachingEngine depends on NarrativeGenerator being stable (EPIC-03) and KnowledgeGapEngine (V1.1, already available). Resource library governance (ADR-030) is also a prerequisite. |
| **EPIC-05** | Evidence Freshness extends ObservationStore with decay semantics. Placed after EPIC-03/04 to avoid TTL interference during early pipeline validation. Can be parallel with EPIC-04 if ObservationStore interface is stable. |
| **EPIC-06** | Calibration Framework validates the complete pipeline (evaluation + language + profile). Requires EPIC-00 (LanguageConfig) and a stable evaluation pipeline. Runs in CI; does not block feature development. |
| **EPIC-07** | Enterprise Extensibility is additive-only (TenantContext placeholder). Placed last as it depends on SessionHistory schema being stable (EPIC-02/03) and introduces zero functional behaviour. |

### Parallel Tracks (within Phase)

The above order defines the dependency sequence. Within each phase, parallel execution is permitted where dependencies allow:

| Phase | Parallel Tracks |
|---|---|
| Phase 1 | EPIC-00 + ADR-016/ADR-017 preparation + ADR-022 SessionHistory schema |
| Phase 2 | EPIC-01 + EPIC-02 (after EPIC-01 interface frozen) |
| Phase 3 | EPIC-03 + SessionHistory persistence implementation |
| Phase 4 | EPIC-04 + EPIC-05 + EPIC-07 |
| Phase 5 | EPIC-06 + Cost Optimisation + V1.2 release gate |

## SECTION B3 — Mandatory Workflow

### Frozen Workflow

```
Architecture
    ↓
Contracts
    ↓
ADR
    ↓
Implementation
    ↓
Tests
    ↓
Audit
    ↓
Freeze
    ↓
Certification
```

### Workflow Definitions

| Stage | Definition | Authority |
|---|---|---|
| **Architecture** | Domain boundaries, aggregates, invariants, and extension points are designed. No code exists. | Architecture Certificate (this document) |
| **Contracts** | Pydantic schemas, ADCs, function signatures, and interface definitions are frozen. `extra=forbid`, `schema_version` applied. | ADR |
| **ADR** | An Architecture Decision Record is authored and accepted before implementation begins for any new component. | Platform Engineering Manifest |
| **Implementation** | Code is written to express the frozen contracts. No deviation from contracts is permitted. | Implementation Gate |
| **Tests** | Unit, integration, and behavioural tests are written. Coverage gates must pass before merge. | Quality Gate |
| **Audit** | An architectural audit verifies no boundary violations, no hidden coupling, no second writers. | Audit Gate |
| **Freeze** | Component API is declared frozen. No breaking changes without a new ADR. | Freeze Gate |
| **Certification** | Milestone is certified. Frozen components are registered in INDEX. | Certification Gate |

## SECTION B4 — Implementation Rules

### Frozen Rules

1. **No architectural shortcuts.** Every implementation decision that touches an architectural boundary requires an ADR. Speed is not justification.

2. **No runtime decisions that violate ADRs.** If an accepted ADR says "X is the sole writer of Y", no runtime path may write Y from outside X.

3. **No duplicated ownership.** Each aggregate, concept, and contract has one owner. Implementation must never introduce a second owner under any condition (error path, test convenience, migration utility).

4. **No second writers.** Single-writer invariants are implementation invariants. A "convenience write" is a defect. A "migration script write" is a defect if it bypasses the sole writer.

5. **No mutable history.** SessionHistory, Observations, EvidenceSignals, Narratives, and CoachingPlans are write-once. No update path is permissible. Forward-compatible additive schema changes require an ADR.

6. **No hidden coupling.** Dependencies must be explicit, directional, and declared at the architecture level. Circular imports, shared mutable state, and implicit side-effects are forbidden.

7. **No business logic inside infrastructure.** `infrastructure/` contains configuration, storage adapters, and execution runtimes. Domain rules, invariant enforcement, and scoring logic belong in `domain/` or `services/`.

8. **No infrastructure inside domain.** `domain/` contains contracts, policies, and events. No SQLAlchemy models, no file I/O, no external service calls inside `domain/`.

## SECTION B5 — Implementation Gates

### Before Coding

**Gate purpose:** Confirm that architecture and contracts are ready before writing any implementation code.

**Mandatory checks:**
- [ ] ADR for the component is accepted.
- [ ] Pydantic contract is frozen (`extra=forbid`, `schema_version` assigned).
- [ ] Single-writer identity is documented.
- [ ] Extension points are identified and will not be bypassed.
- [ ] No upstream unfrozen contract dependency.

---

### Before Merge

**Gate purpose:** Confirm that implementation matches architecture before the branch is merged.

**Mandatory checks:**
- [ ] All unit tests pass.
- [ ] Coverage ≥ 90% for the implemented module.
- [ ] No new circular imports.
- [ ] No contract loosening (`extra=forbid` preserved; no fields removed).
- [ ] Single-writer invariant verified (no second writer introduced).
- [ ] Linter passes without suppression.
- [ ] ADR compliance verified (implementation matches the accepted decision).

---

### Before Release

**Gate purpose:** Confirm that the complete milestone deliverable is production-ready.

**Mandatory checks:**
- [ ] Full test suite passes (zero failures).
- [ ] Calibration CI gate active and passing (EPIC-06 onwards).
- [ ] All frozen contract files present in `domain/contracts/`.
- [ ] Architecture audit passed (no boundary violations, no hidden coupling).
- [ ] Replay verified for sessions produced in this milestone.
- [ ] No open P0/P1 findings.
- [ ] INDEX.md updated with milestone frozen baseline.

---

### Before Freeze

**Gate purpose:** Confirm that a component is ready to be declared frozen (no future breaking changes without ADR).

**Mandatory checks:**
- [ ] Public API matches the contract specification.
- [ ] All acceptance criteria from the corresponding Epic satisfied.
- [ ] Regression test suite covers all known edge cases.
- [ ] Component is registered in the INDEX frozen baseline table.
- [ ] ADR governing this component is marked ACCEPTED in INDEX.
- [ ] Documentation in TDS is current.

## SECTION B6 — Implementation Quality

### Frozen Quality Criteria

| Criterion | Standard |
|---|---|
| **Coverage** | ≥ 90% unit test coverage per module. New detectors and service components require 100% happy-path coverage. |
| **Regression** | Full regression suite must pass before any Epic is marked complete. Zero tolerance for P0/P1 regression. |
| **Performance** | No single detector may exceed 5ms execution time (ADR-054). FeatureEngine full cycle ≤ 20ms. |
| **Determinism** | All non-LLM components must be deterministic (same input → same output). Clock-based components must accept a mockable clock interface. |
| **Documentation** | Every new component must have a contract file in `domain/contracts/`, a governing ADR registered in INDEX, and TDS section updated. |
| **Compatibility** | New schema versions must be backward-compatible. Existing `SessionHistory` records must remain replayable after schema additions. |
| **Replay** | All sessions produced in a milestone must be replayable via `ReplayEngine` after that milestone. Non-replayable sessions are a P0 defect. |
| **Auditability** | All components that write to a frozen aggregate must emit a traceable event or log entry (ADR-041, ADR-047). |

## SECTION B7 — Future Methodology Hooks

The following methodology reviews are introduced conceptually as placeholders. They will become formal engineering methodology practices in the "Thinking Like an AI Engineer" framework.

### Architecture Review

**Placeholder definition:** A structured review of architectural decisions, boundary definitions, and ADR compliance performed at key milestones.  
**Trigger:** Before implementation starts on any Epic that introduces a new Aggregate Root, a new Layer, or a new Extension Point.  
**Not yet active:** Methodology details are deferred to a future Engineering Methodology document.

---

### Implementation Review

**Placeholder definition:** A structured review of implementation code against the Architecture Certificate and frozen ADRs.  
**Trigger:** Before any component is declared frozen.  
**Not yet active:** Methodology details are deferred to a future Engineering Methodology document.

---

### Engineering Review

**Placeholder definition:** A combined review of architecture compliance, code quality, test coverage, and performance budget adherence.  
**Trigger:** Before any major release (milestone boundary or version gate).  
**Not yet active:** Methodology details are deferred to a future Engineering Methodology document.

---

### Release Review

**Placeholder definition:** A final review of the complete milestone deliverable — architecture certification, test results, regression, documentation, and release readiness.  
**Trigger:** Before any V1.x release tag is created.  
**Not yet active:** Methodology details are deferred to a future Engineering Methodology document.

---

**These four review types are conceptual anchors only.** They establish the vocabulary and intent for a rigorous engineering methodology without prescribing process mechanics. The full methodology will be codified in a dedicated document as the "Thinking Like an AI Engineer" framework matures.
