## RP-01 — V1.2 Architecture Retrospective

---

### Executive Summary

V1.2 migrated a legacy V1.1 runtime — built around a monolithic `InterviewEvaluation` object and unstructured LLM calls — into a structured knowledge pipeline ending in an immutable, queryable `Report`. The migration took nine major increments (MIG-01 through MIG-09) plus four RC1 corrections. The final architecture is coherent, ownership is clear, and all PATs are satisfied. The retrospective identifies three categories of outcome: decisions that worked exactly as intended, decisions that required mid-migration correction, and patterns that were not designed but emerged and should be formalized.

The single most important architectural lesson: **immutability and single ownership, enforced at the contract level, prevented every class of data corruption encountered in V1.1.** The biggest mistakes were all forms of dual ownership introduced under migration pressure — temporary bridges, compatibility layers, and fallback paths that outlived their intended scope.

---

### 1. Biggest Successes

**ObservationStore as the extraction boundary**

The decision to route all observation data through a single, session-scoped `ObservationStore` before any pipeline processing was the most consequential structural decision in V1.2. It created a hard boundary between raw signal extraction and structured feature derivation. Nothing downstream of the store depends on how observations were produced. This made `KnowledgePipeline` testable in isolation, enabled deterministic replay contract definitions, and prevented the signal leakage that had caused inconsistent `EvidenceStore` state in V1.1.

**`CandidateProfileBuilder` as sole construction path**

Enforcing a single builder for `CandidateProfile` — with `frozen=True` on the result — eliminated an entire class of bugs where V1.1 profile updates were partially applied. Because the builder is the only construction path, every `CandidateProfile` is structurally valid by construction. The profile can be passed freely between nodes without defensive copies. This pattern proved correct repeatedly and became the template for `ReportBuilder`, `SessionHistoryBuilder`, and `KnowledgeSnapshotBuilder`.

**Cascading closure: SessionHistory → Report**

Structuring the session-end as a cascade (`session_close_node` → `SessionHistory` → `report_node` → `Report`) rather than a single monolithic close operation separated concerns naturally. `session_close_node` owns knowledge assembly; `report_node` owns report projection. Each is independently testable, independently non-fatal, and independently idempotent. The idempotency guard on `session_close_node` (return immediately if `session_history is not None`) proved valuable in practice — it made the node safe to retry without side effects.

**LangGraph as the sole orchestrator (PAT-06)**

Formalizing LangGraph as the only runtime orchestrator — and enforcing this with a startup invariant (`RuntimeError` on unconfigured `navigation_node`) — prevented a class of architectural drift that had appeared in V1.1: sequential coordinator objects that duplicated routing logic. Once PAT-06 was declared and enforced, no new orchestration path was introduced. The enforcement mechanism (fail-fast `RuntimeError` instead of silent fallback) was the correct choice: it surfaced configuration errors at startup, not at question five of a live interview.

**TCP fields as migration scaffolding**

The Temporary Construction Placeholder pattern (PAT-04) — adding V1.2 fields to `InterviewState` as nullable with no V1.1 reader — allowed each migration increment to be deployed independently without breaking the live runtime. This was the structural mechanism that made the migration safe. The nullable contract meant that every migration increment could be tested against the prior runtime before being activated.

**`InterviewMemory` and session_metrics persistence**

The decision to make `InterviewMemory` the single accumulation point for reasoning intelligence, with clearly defined fields (`evidence_store`, `coverage_state`, `reasoning_history`, `session_metrics`), created a clean observable record of reasoning state. When RC1-03 discovered that `session_metrics` was silently discarded on reconstruction, the fix was surgical: the problem was located immediately because `InterviewMemory` has one writer (`reasoner_node`) and one update path (`ReasonerService`). Dual ownership would have made this undetectable.

### 2. Biggest Mistakes

**Dual `CandidateProfile` ownership (corrected in MIG-03/RS-02)**

The original V1.2 design briefly allowed two separate paths to produce a `CandidateProfile`: the legacy `candidate_profile_engine` path and the new `FeatureEngine` path. This created a period where neither path was authoritative. The mistake happened because the migration was designed incrementally with parallel paths, and the decision to deprecate the legacy path was not formalized early enough. Detection occurred during CAR-6 when the ownership review found two producers. Correction required deleting the legacy path files entirely (`base_updater.py`, `candidate_profile_engine.py`, `candidate_profile_factory.py`, `coverage_updater.py`, `dimension_trace_updater.py`, `trend_updater.py`). **Lesson:** Never allow two production paths to produce the same artifact, even temporarily. The migration scaffolding should disable the legacy path immediately, not defer deprecation.

**`InterviewMemoryContext` as a transitional bridge (corrected in RC1-02)**

`InterviewMemoryContext` was introduced as a compatibility layer between the V1.1 memory model and the V1.2 `InterviewMemory`. It survived far beyond its intended scope, became a dependency of `AdaptiveInterviewMemoryBridge`, and was being updated by `question_node` in parallel with `InterviewMemory`. This created two diverging memory representations for the same session. The mistake happened because "temporary bridge" objects are difficult to remove once they acquire dependents. Detection required a full ADR-032 compliance audit. Correction required deleting three files and inlining the bridge logic. **Lesson:** Bridges must have a hard deletion date declared at creation. Any object labeled "temporary" that acquires more than one consumer has escaped its scope.

**Legacy navigation fallback (corrected in RC1-01)**

`_legacy_navigation_node()` in `navigation_node.py` was a silent fallback that activated when `configure_navigation_node()` had not been called. This meant the node could silently change its orchestration strategy at runtime without any observable error. The fallback survived through multiple architecture reviews because it was never invoked in production — but it remained a latent path that contradicted PAT-06. Detection required targeted inspection of `navigation_node.py`. Correction was removing the function entirely and replacing the conditional with a `RuntimeError`. **Lesson:** Silent fallbacks are architecture violations. Any code path that changes behavior without emitting an observable error is a hidden orchestrator. Fail-fast invariants should be the default, not the exception.

**Close-time recomputation (corrected across MIG-04 through MIG-08)**

The V1.1 design ran `FeatureEngine`, `NarrativeGenerator`, and `CoachingEngine` at session close, recomputing everything from scratch using the full transcript. This meant session close was slow, non-deterministic (LLM calls), and expensive. The decision to move computation into the reasoning cycle (`reasoner_node`) and make session close a pure assembly step was correct — but it required seven migration increments to fully implement. The migration took longer than necessary because the original architecture had no clear separation between "computation produces knowledge" and "session close assembles knowledge." **Lesson:** The boundary between computation and assembly must be a first-class architectural decision, not an emergent one. V1.3 should establish this boundary formally before the first migration increment.

**`session_metrics` persistence gap (corrected in RC1-03)**

`ReasonerService.reason()` updated `session_metrics` but `reasoner_node` reconstructed `InterviewMemory` without including the updated metrics. The fix required adding `session_metrics` to the return tuple of `reason()`. This mistake happened because the reconstruction path in `_append_reasoning_entry` was written before `session_metrics` was added to `InterviewMemory`, and the field was never threaded through the return path. Detection required a CAR-7 field-level audit. **Lesson:** Every field added to an immutable reconstruction target must be explicitly threaded through all return paths at the time of addition. Reconstruction paths should be reviewed as a checklist, not as an afterthought.

**`report_output` as a V1.1 residue field**

`InterviewState.report_output: str | None` was a V1.1 convenience field that held the rendered HTML string of the report. It was never removed when the V1.2 `Report` contract was introduced. As a result, RC1-04 discovered that the UI was routing to `UIState.REPORT` on `interview_evaluation is not None` rather than on `state.report is not None`. The field still exists as a dead declaration. **Lesson:** When a new artifact supersedes an old one, the old field should be marked `deprecated` or removed in the same migration increment. Leaving it creates invisible routing ambiguity.

### 3. ADR Lessons

**Most valuable by architectural impact:**

**ADR-032 (Single memory model)** — The most impactful ADR by a significant margin. By declaring `InterviewMemory` as the sole runtime memory representation and mandating deletion of `InterviewMemoryContext`, it prevented the divergence from becoming permanent. The lesson: ADRs that declare a deletion are more valuable than ADRs that declare an addition. Deletion ADRs force migration completion.

**ADR-022 (KnowledgeSnapshot immutable closure)** — Structuring `KnowledgeSnapshot` as an immutable assembly of all session knowledge artifacts at close time created the foundation for deterministic replay. Without this ADR, `session_close_node` would have had no clear contract and the cascading closure pattern could not have emerged. The "closure" framing — the idea that a session close is a point-in-time seal, not an ongoing computation — was the most useful conceptual contribution of the V1.2 ADR set.

**ADR-020 (Feature lifecycle)** — Defining `FeatureUpdater`, `FeatureComposer`, `FeatureMergePolicy`, and `FeatureReplacementPolicy` as separate interfaces kept the feature computation extensible without requiring changes to `FeatureEngine`. Every subsequent addition of a new feature type required only a new `FeatureUpdater` implementation — no engine changes. This is the definition of a successful extensibility contract.

**ADR-038 (Session metrics persistence)** — Small in scope but revealed a systematic gap: the absence of a formal contract between what `ReasonerService` computes and what `reasoner_node` persists. The ADR documenting this bug has more preventive value than its fix — it established the principle that every computed field must be explicitly declared in the persistence contract.

**ADR-052 (EvaluationSignal bridge)** — Correctly deferred rather than blocked MIG-02 by bridging V1.1 `QuestionResult` signals into V1.2 `ObservationStore`. Its value is not in the bridge itself but in its explicit classification: a temporary bridge with a defined scope. The naming convention (`ADR-052` rather than a MIG ticket) elevated it to an architectural decision rather than a workaround. This distinction matters.

**ADRs that proved less valuable:**
ADR-023 (Narrative) and ADR-025 (Coaching) — both correct and implemented — were straightforward enough that a PAT reference would have been sufficient. They generated more documentation than architectural guidance.

---

### 4. PAT Lessons

**Most valuable PATs:**

**PAT-06 (LangGraph sole orchestrator)** — Prevented the most dangerous class of architectural drift: hidden orchestration. Every migration sprint produced pressure to add "just one coordinator" or "just one service that calls two nodes in sequence." PAT-06 made that pressure visible and refusable. The startup invariant (`RuntimeError`) turned this from a documentation rule into an enforced contract. **Should evolve:** add a corollary for service-level orchestration — services may not call other services in a way that implements routing logic.

**PAT-02 (Single State Owner)** — The `InterviewState` model was never forked. Every node received the same state object and returned a new copy. This prevented the state divergence that had made V1.1 debugging extremely difficult. The pattern's value was realized most clearly during the session_metrics bug — because there was exactly one state object, the persistence gap was immediately locatable.

**PAT-05 (Single Builder per artifact)** — Enforcing this throughout the migration prevented the dual-path problem from recurring after MIG-03. Every new immutable artifact — `KnowledgeSnapshot`, `SessionHistory`, `Report` — was assigned a single builder at creation. No artifact acquired a second construction path.

**PAT-04 (TCP fields)** — The most migration-specific PAT and the least likely to be reused in V1.3. Its value was real but bounded: it enabled safe incremental activation of new fields. The mechanism — nullable fields with no V1.1 reader — worked because the team was disciplined about the "no V1.1 reader" constraint. If any V1.1 node had read a TCP field before it was activated, the migration would have silently produced wrong state.

**PATs that should evolve:**

**PAT-01 (Immutable State Transitions)** should add an explicit prohibition on state mutation outside of `model_copy`. Currently the pattern specifies what to do; it should also specify what is forbidden, with examples of the exact patterns that violate it.

**A new PAT is warranted:** "Reconstruction Completeness" — any function that reconstructs an immutable object from a prior version must explicitly declare every field it copies, and the declaration must be updated whenever a field is added to the target type. RC1-03 revealed that no such pattern existed, which is how `session_metrics` could be silently dropped.

### 5. Patterns That Emerged Naturally

**Cascading Closure**

Observed independently in `session_close_node → report_node`. The pattern: session completion is a cascade of idempotent steps, each consuming the output of the prior step, each write-once. Not designed at the start — emerged from the requirement that `report_node` must be independently non-fatal. Should become an official pattern for any future multi-step completion pipeline (e.g., a future persistence node after `report_node`).

**Projection Artifact**

`CandidateProfileSnapshot` is a direct projection of `CandidateProfile.features` at close time — no computation, no new data, just a point-in-time view for historical reference. `Report` similarly projects `SessionHistory` into a presentation-ready artifact. This pattern — an immutable projection of a mutable runtime state, captured at a lifecycle boundary — appeared three times independently. Should be formalized: a Projection Artifact is created at a lifecycle boundary, contains no new computation, and is write-once.

**Runtime First, Projection Later**

All computation happens during the live runtime cycle (in `reasoner_node`). Session close is projection only. Report generation is projection only. This principle was violated by V1.1 (close-time recomputation) and corrected by V1.2. It should be explicitly formalized: no LLM call, no `FeatureEngine` execution, no `KnowledgePipeline` invocation may occur outside the live reasoning cycle.

**Sole-Writer Nodes**

Several nodes declare themselves as the sole writer of a specific field: `session_close_node` for `session_history`, `report_node` for `report`, `reasoner_node` for `observation_store` and `candidate_profile_v2`. This pattern appeared in every new node written for V1.2, but was never named. Should become a structural node documentation standard: every node must declare its owned write targets.

**Non-Fatal Node with Flag Clearing**

`report_node` and `session_close_node` both follow the pattern: attempt the primary operation; on any failure, log a warning and clear the processing flags anyway. This ensures the UI is never stuck in a loading state regardless of backend failure. The pattern emerged independently in both nodes. Should be formalized for any node that clears `is_processing`.

### 6. Validated Principles

**Proved correct:**
- Immutability prevents a class of bugs that defensive copying cannot prevent. `frozen=True` on domain contracts was the right default.
- Single ownership at the contract level is more valuable than documentation of ownership intent.
- Fail-fast at startup is always preferable to silent fallback. Every silent fallback in V1.1 eventually caused a production incident or architectural confusion.
- Deleting code is more valuable than deprecating it. Every "deprecated" object in V1.1 that was not deleted eventually acquired a new consumer.
- Architectural review after migration completion is worth the investment. CAR-6 and CAR-7 produced more actionable findings than continuous per-sprint reviews.

**Proved incorrect:**
- "Parallel paths during migration are safe if the legacy path is never activated." MIG-03 demonstrated that a parallel path, even if unused, creates confusion about the authoritative path and adds maintenance cost. There was no situation in V1.2 where a parallel path was simultaneously safer and cheaper than a direct cutover.
- "Bridges can be removed lazily after the migration is stable." `InterviewMemoryContext` and `AdaptiveInterviewMemoryBridge` required dedicated sprint time to remove. The cost of removal did not decrease with time — it increased as tests accumulated.

**Assumptions that changed:**
- Initial assumption: session close should be the authoritative point for profile computation. Final position: session close is a projection of runtime-accumulated knowledge. The migration required seven increments to complete this reversal — not because the implementation was difficult, but because the conceptual reversal took time to fully commit to.
- Initial assumption: the UI should own report rendering and could safely read any state field. Final position: the UI is a reader of `state.report` only. The routing gate (`UIStateMachine` checking `state.report is not None`) formalized the boundary.

---

### 7. What Would Be Designed Differently Today

**Declare the computation/projection boundary on day one.**
The most expensive architectural work in V1.2 was moving computation out of session close and into the reasoning cycle. If the principle "the runtime cycle is the only computation site; closure is projection only" had been declared as an architectural constraint before MIG-01, the seven migration increments addressing this reversal would not have been necessary.

**Assign sole ownership to every field in `InterviewState` at field creation.**
V1.1 fields accumulated without ownership declarations. V1.2 added TCP annotations, but the ownership declaration was not required at field creation — it was retrofitted during CAR-6. A field should not exist in `InterviewState` without a declared sole writer and declared readers.

**No parallel production paths, ever.**
Every case of "we'll keep the old path until the new one is stable" produced technical debt that required a dedicated removal sprint. The correct migration strategy is: add the new path as the sole path behind a feature flag, not alongside the old path.

**Make `InterviewMemory` reconstruction a declared contract, not an implementation detail.**
The `_append_reasoning_entry` function in `reasoner_node` should have been accompanied by a checklist: "all fields of `InterviewMemory` must be explicitly passed through this function." Without the checklist, `session_metrics` was silently dropped for several sprints before CAR-7 detected it.

**Formalize the cascading closure pattern before the first session-close migration.**
`session_close_node` and `report_node` independently reinvented the same "idempotent step in a closure cascade" pattern. If the pattern had been declared before MIG-04, both nodes would have been implemented consistently from the start.

### 8. RC2/RC3 Assessment

**RC2 items — each correctly deferred:**

- `report_output` dead field: correctly deferred. Removing it at RC1 would have required a state migration that added risk without adding capability.
- MIG-03B fallback hardening: correctly deferred. The fallback is inert for all new sessions. Hardening it to `RuntimeError` is RC2 hygiene, not a correctness fix.
- Registry increment test consolidation: correctly deferred. No production impact.
- `InterviewStateMapper._aggregate_weaknesses()`: correctly deferred pending confirmation it is unreachable.

**RC3 items — each correctly deferred:**

- Scoring pipeline migration (move `InterviewEvaluation` data into `Report`): correctly deferred. This requires a new ADR, a new migration sequence, and potential changes to `ReportBuilder`. Doing it at RC1 would have destabilized the presentation layer.
- `ReplaySession` orchestration: correctly deferred. The contract exists; the business need is not yet validated. Implementing the node before the need is confirmed would produce another MIG-series of cleanup work.
- Retiring `InterviewEvaluation` entirely: correctly deferred until RC3 scoring migration completes.

**One item that may have been deferred too aggressively:**
The `report_output` field could have been removed during RC1-04 since RC1-04 already touched all the same presentation paths. The marginal cost of removing it was low at that moment. This is a minor observation, not a correctness concern.

### 9. Architectural Recommendations for V1.3

**Formalize the Scoring Pipeline as a V1.2-style migration.**
`InterviewEvaluation` is a V1.1 artifact that survives in V1.2 because scoring migration was deferred. V1.3 should begin with a declared boundary: `Report` becomes the authoritative scoring artifact. This requires extending `ReportBuilder` to accept scoring dimensions, updating `FinalReportDTO` to read from `Report`, and removing `interview_evaluation` as a routing or presentation dependency. The migration should follow the V1.2 TCP pattern: add fields to `Report`, activate them one by one, then delete `InterviewEvaluation`.

**Implement the Replay runtime as a closed pipeline.**
The `ReplayMetadata`, `ReplayFeatureEngine`, and `CandidateProfileSnapshot` contracts are complete. V1.3 should add a `replay_node` that consumes a `SessionHistory` and produces a `CandidateProfile` deterministically from stored features. The node must be non-fatal and write-once. It must not activate the live `KnowledgePipeline`. This is the natural extension of the cascading closure pattern.

**Introduce multi-session profile continuity.**
V1.2 `CandidateProfile` is session-scoped. V1.3 should define a `LongitudinalProfile` artifact that accumulates `CandidateProfileSnapshot` instances across sessions. The architecture already has the required inputs: each session produces a `CandidateProfileSnapshot` inside its `KnowledgeSnapshot`. The V1.3 design challenge is ownership: who produces `LongitudinalProfile`, when is it updated, and where is it stored. This must be solved architecturally before any implementation begins.

**Formalize the four emerging patterns as official PATs.**
Cascading Closure, Projection Artifact, Runtime First / Projection Later, and Sole-Writer Node each appeared three or more times independently during V1.2. They are already being followed — they simply lack official names and enforcement mechanisms. Formalizing them before V1.3 begins will prevent the drift that occurred when V1.2 started without PAT-06 being fully enforced.

**Introduce a Reconstruction Completeness PAT.**
Any function that reconstructs an immutable object must carry an explicit declaration of every field it copies. This prevents the session_metrics class of bug from recurring as new fields are added to `InterviewMemory`, `CandidateProfile`, or any future accumulation artifact.

**Declare the computation/projection boundary as a constitutional constraint for V1.3.**
Before the first V1.3 migration increment is written, the following constraint must be declared and enforced: no LLM call, no `FeatureEngine` execution, and no `KnowledgePipeline` invocation may occur outside the live reasoning cycle. Session close is projection. Report generation is projection. Replay is controlled recomputation under explicit flag. Any future node that violates this constraint requires a new ADR to justify the exception.

---

### Final Lessons Learned

1. **Immutability at the contract level is non-negotiable.** Every data corruption bug in V1.1, and every migration complexity in V1.2, traced back to mutable shared state. The `frozen=True` default on all domain contracts was the correct decision; it should be the default for any future project.

2. **Deletion is the most important migration action.** Adding new paths is easy. Removing old ones requires discipline. The cost of a deprecated-but-not-deleted object grows linearly with time. The correct migration completion criterion is "legacy code deleted," not "new code activated."

3. **Ownership must be declared, not assumed.** V1.1 had no ownership model. V1.2 had ownership by convention until CAR-6 formalized it. V1.3 should have ownership declared at field creation, enforced at code review, and verified by architectural tests.

4. **Fail-fast invariants are worth the initial test friction.** The `configure_navigation_node()` `RuntimeError` broke several tests on introduction. The tests were easy to fix. The alternative — a silent fallback that could activate at runtime — would have been far more costly to debug in production.

5. **Architectural reviews after migration milestones are essential.** CAR-6 and CAR-7 each produced actionable findings that per-sprint reviews missed. Migrations introduce drift that accumulates over time and is not visible in any individual increment. Post-milestone reviews are not optional overhead — they are a required part of the migration lifecycle.

6. **The computation/projection boundary is the most important architectural invariant in a knowledge pipeline.** Every expensive migration in V1.2 was ultimately about moving computation out of projection paths. Declaring this boundary before V1.3 begins is the single highest-leverage architectural action available.
