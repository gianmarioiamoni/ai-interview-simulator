## SECTION A1 — Architecture Summary

### V1.2 Complete Architecture

V1.2 transforms the AI Interview Simulator from a session-scoped evaluation engine into a persistent, longitudinal coaching platform. The architecture is organised into six layers, each with a single responsibility and a single-writer ownership model.

---

#### Knowledge Layer

**Purpose:** Produce the structured knowledge model for a candidate's session — converting raw evaluation signals into a typed, versioned, decayable evidence graph.

| Component | Role |
|---|---|
| `ObservationExtractor` | Transforms `EvidenceSignal` → typed `Observation` (sole writer) |
| `ObservationStore` | Append-only, independent Aggregate Root; enforces freshness TTL; decays stale observations |
| `ObservationType` | Controlled vocabulary; static, domain-level enum |
| `FeatureEngine` | Orchestrates FeatureUpdaters; sole producer of all `ProfileFeature` values |
| `FeatureUpdater` (family) | `ObservationUpdater`, `ReplayUpdater`, `CalibrationUpdater` — each updates a specific feature dimension |
| `ProfileFeature` | Derived, versioned, language-agnostic behavioral trait; owned by `CandidateProfile` |
| `CandidateProfile` | Consumes `ProfileFeature[]` only; independent Aggregate Root |

**Invariants:** ObservationStore is append-only. ProfileFeatures are always derived by FeatureEngine. No direct external writes to CandidateProfile.features.

---

#### Persistence Layer

**Purpose:** Provide durable, forward-compatible, immutable session storage that is the single source of truth for replay, progress tracking, and calibration.

| Component | Role |
|---|---|
| `SessionHistory` | Write-once Aggregate Root; stores complete session snapshot (transcript, evaluations, Narrative, CoachingPlan, LanguageProfile) |
| `KnowledgeSnapshot` | Versioned snapshot of `CandidateProfile` at session close |
| `CandidateProfileSnapshot` | Strategy for snapshotting profile state without live aggregate dependency |
| `KnowledgeEpoch` | Temporal boundary marker between sessions for cross-session trend computation |
| `CandidateIdentity` | Ownership anchor for `SessionHistory`; enables cross-session aggregation |
| `LearningProgress` | Cross-session derived view; computed read-only from `SessionHistory` |

**Invariants:** SessionHistory written once; never mutated. Schema additions are additive only. CandidateIdentity is the ownership anchor.

---

#### Action Layer

**Purpose:** Convert the candidate knowledge model into actionable coaching output.

| Component | Role |
|---|---|
| `KnowledgeGapEngine` | Derives `KnowledgeGap[]` from evaluation results; ranked by severity |
| `CoachingEngine` | Reads `KnowledgeGap[]` + `ProfileFeature[]`; produces `CoachingPlan` |
| `CoachingPlan` | Write-once Aggregate Root; contains ranked `CoachingAction[]` + `LearningObjective[]` |
| `CoachingAction` | Leaf node; maps 1:1 to a `KnowledgeGap` |
| `StudyRecommendation` | Actionable learning directive; gap + resources + estimated study time |
| `ResourceLibrary` | Curated, version-controlled resource catalog governed by ADR-030 |

**Invariants:** Every `CoachingAction` must reference a `KnowledgeGap`. CoachingEngine never reads raw EvaluationResults directly.

---

#### Language Layer

**Purpose:** Enforce language independence — no evaluation, generation, or execution component may hardcode assumptions about a specific programming language.

| Component | Role |
|---|---|
| `ProgrammingLanguage` | Abstract domain concept; Python/JS/TS are concrete implementations |
| `LanguageRegistry` | Static registry of all supported language implementations |
| `LanguageConfig` | Session-scoped language configuration (ADR-019) |
| `LanguagePolicy` | Evaluation rules per language; does not alter dimension weights |
| `LanguageExecutor` | Abstracted runtime dispatch interface (ADR-027) |
| `LanguageProfile` | Complete session language configuration; immutable within a session |
| `LanguageSelectionPolicy` | Algorithm for mixed-mode (50/50) language selection (ADR-028) |

**Invariants:** EvaluationDimension weights are language-independent. EvidenceSignal semantics are language-agnostic. LanguagePolicy never modifies the evaluation rubric structure.

---

#### Resource Layer (Narrative + Replay)

**Purpose:** Transform the knowledge model and coaching output into candidate-facing artifacts, and provide faithful reconstruction of any completed session.

| Component | Role |
|---|---|
| `NarrativeGenerator` | Profile-feature-aware coaching text (ADR-023); reads `CandidateProfile` + `KnowledgeGap[]` |
| `Narrative` | Write-once aggregate: `NarrativeSection[]` + `NarrativeInsight[]` |
| `NarrativeInsight` | Atomic coaching unit; must be anchored to Observation or ProfileFeature |
| `ReportBuilder` | Assembles `Narrative` + `CoachingPlan` → final report; writes to `SessionHistory` |
| `ReplayEngine` | Read-only reconstruction from `SessionHistory`; zero LLM calls |

**Invariants:** NarrativeInsights must be evidence-anchored. ReplayEngine never invokes LLM calls. NarrativeGenerator never writes Observations.

---

#### Governance

**Purpose:** Ensure scoring integrity, cost control, and enterprise extensibility without contaminating domain logic.

| Component | Role |
|---|---|
| `CalibrationProfile` | Expected score distribution per (role, seniority, difficulty, language) |
| `CalibrationFramework` | CI gate; validates scoring constants; never modifies evaluation constants |
| `TenantContext` | Additive-only placeholder (nullable V1.2); V2 multi-tenant migration hook |
| `CostOptimisationFramework` | Model routing, token budgeting, per-call telemetry |

**Invariants:** CalibrationProfile is a validation artifact, not a tuning mechanism. TenantContext is a no-op in V1.2. Cost optimisation never affects evaluation reasoning quality.

## SECTION A4 — Protected Extension Points

Extension points are architectural hooks reserved for future features. Future implementations MUST extend through these points, never by modifying existing frozen components.

| Extension Point | Governing ADR | How to Extend |
|---|---|---|
| **New ProgrammingLanguage** | ADR-019, ADR-027 | Register in `LanguageRegistry`; implement `LanguageExecutor`; author `LanguagePolicy`; add question stubs. Zero domain change. |
| **New ObservationType** | ADR-016 | Add to controlled vocabulary. `ObservationExtractor` and `ObservationStore` require no change. |
| **New FeatureUpdater** | ADR-020 | Implement the `FeatureUpdater` protocol; register in `FeatureEngine`. No change to `CandidateProfile` interface. |
| **New ProfileFeature** | ADR-018 | Add to taxonomy with schema version. Existing features untouched. `NarrativeGenerator` consumes via feature-key lookup. |
| **New PatternDetector** | ADR-051, ADR-053 | Implement `PatternDetector` ABC; register with priority in `PatternDetectorRegistry`. Zero changes to pipeline. |
| **Cross-session ProfileFeature accumulation** | ADR-018, ADR-021 | Activate `ReplayUpdater` in `FeatureEngine`. No schema change — reserved fields already present. |
| **TenantContext (V2 multi-tenant)** | ADR-016A, ADR-022 | Populate `tenant_id` fields (already in schema as nullable). No schema migration required. |
| **CoachingEngine ranking strategy** | ADR-025 | Replace or extend ranking algorithm via configuration. `CoachingPlan` interface unchanged. |
| **ResourceLibrary curation** | ADR-030 | Add/update resources following governance policy. Library is a version-controlled artifact, not code. |
| **CalibrationProfile baseline update** | ADR-031 | Author new baseline following governance policy. No code change unless evaluation constants change. |

**Rule:** Future features extend by registering with an existing extension point, not by modifying frozen aggregates or bypassing single-writer invariants.

## SECTION A5 — Remaining ADRs

### ADR-024 — Calibration Framework CI Gate Design

**Classification:** Support ADR  
**Scope:** Engineering governance — defines when a calibration run blocks a CI merge.  
**Not Core Architecture because:** It governs an operational concern (CI gate configuration) rather than defining a domain boundary, aggregate, or runtime flow. The `CalibrationProfile` and `CalibrationFramework` are architecturally frozen by the V1.2 architecture. ADR-024 specifies the CI integration detail.  
**Status:** Pending — required before EPIC-06 implementation starts.

---

### ADR-029 — Enterprise Extensibility — TenantContext Placeholder Design

**Classification:** Support ADR  
**Scope:** V2 migration hook — defines how `TenantContext` is inserted as a nullable, no-op field in V1.2 session state and storage schema.  
**Not Core Architecture because:** It does not define a new aggregate, flow, or domain boundary. It specifies the additive schema reservation that enables V2 multi-tenant migration without a full rewrite. The architectural principle (additive-only, no-op in V1.2) is already codified in ADR-016A and ADR-022.  
**Status:** Pending — required before EPIC-07 implementation starts.
