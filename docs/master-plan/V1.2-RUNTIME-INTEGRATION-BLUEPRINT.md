### Reasoner flow direction (corrected vs. initial intuition)

The correct integration direction is:

```
Evaluation → EvaluationSignalWriter → EvidenceStore
                                           ↓
                                    ReasonerService
                                    (PatternDetectors → PatternMatch[])
                                    (generates EvidenceSignal[] from patterns)
                                           ↓
                                    ObservationExtractor
                                    (reads ALL EvidenceSignals from EvidenceStore)
                                           ↓
                                    ObservationStore
                                           ↓
                                    FeatureEngine → CandidateProfile V1.2
```

**NOT the alternative:**

```
Evaluation → ObservationExtractor → FeatureEngine → CandidateProfile → Reasoner
```

**Rationale:**  
`ObservationExtractor` input is `EvidenceSignal[]` (contract: `ObservationExtractionContext.signals`). `EvidenceSignal[]` is produced by two sources: `EvaluationSignalWriter` (EVALUATION-source) and `ReasonerService._propagate_evidence()` (PATTERN_DETECTOR-source). Both sources must be present in `EvidenceStore` before `ObservationExtractor` runs, because `ObservationStore` must carry the full behavioral signal set — not just evaluation scores.  

If `ObservationExtractor` ran before `ReasonerService`, it would process only EVALUATION-source signals and miss all 13 pattern-detector behavioral signals (confidence calibration, reasoning depth, leadership, adaptability, etc.). The ProfileFeatures derived from such a partial observation set would be structurally incomplete and would fail the evidence-anchoring invariant (I-15).  

`ReasonerService` must run first. `ObservationExtractor` runs after `ReasonerService` in the same `reasoner_node`. The RIR was correct.

## SECTION 6 — Reasoner Future Role (Validated)

### Verdict: **C — Decision Engine** + retained detection-layer **Knowledge Producer**

The RIR classification is confirmed. No correction needed.

### Decomposed responsibilities (post-migration)

| Responsibility | V1.1 | V1.2 end state |
|---|---|---|
| Pattern detection (PatternDetectorRegistry dispatch) | **Owned** | **Retained** |
| EvidenceSignal generation (PATTERN_DETECTOR-source) | **Owned** | **Retained** — these signals feed ObservationExtractor |
| CandidateProfile production (dimension_scores) | **Owned** | **Removed** (MIG-06) |
| ReasonerDecision emission (next_action, follow_up, navigation) | **Owned** | **Retained** — irreplaceable LangGraph branching signal |
| ReasoningTrace (internal audit) | **Owned** | **Retained** |
| Input to ObservationExtractor | Not yet | **New** — ReasonerService output (EvidenceSignal[]) feeds ObservationExtractor |
| Consumer of CandidateProfile V1.2 | Not yet | **New** (MIG-06) — reads `candidate_profile_v2` as context for next cycle |

### Why Reasoner must run before ObservationExtractor

`ObservationExtractor` input is `EvidenceSignal[]`. The complete set of signals for any cycle is:
1. EVALUATION-source signals (from `EvaluationSignalWriter`) — score-derived (KNOWLEDGE_GAP, SHALLOW_ANSWER, REASONING_GAP)
2. PATTERN_DETECTOR-source signals (from `ReasonerService._propagate_evidence()`) — behavioral (confidence, reasoning depth, leadership, adaptability, collaboration, etc.)

If `ObservationExtractor` runs before `ReasonerService`, it processes only category (1). The `ObservationStore` receives no behavioral signals. `FeatureEngine` computes degraded ProfileFeatures. `NarrativeGenerator` produces evidence-incomplete Narratives, violating invariant I-15.

**The sequence within `reasoner_node` is therefore fixed:**
```
EvaluationSignalWriter → EvidenceStore
         ↓
ReasonerService.reason()  [PatternDetectors + CandidateProfileEngine (V1.1, transient)]
         ↓  (EvidenceStore now contains EVALUATION + PATTERN_DETECTOR signals)
ObservationExtractor → ObservationStore
         ↓
KnowledgePipeline → FeatureEngine → CandidateProfile V1.2
```

## SECTION 9 — Pattern Observations

### Emerging Pattern Candidate: **Staged Atomic Artifact Assembly**

**Description:** A single LangGraph node orchestrates multiple independent domain services in a fixed sequence. Each service reads the output of the prior service as its input. All produced artifacts are committed atomically to a single write-once aggregate (`SessionHistory`) at node exit. No artifact is persisted until all are ready. The node itself holds no business logic — it is purely a sequencer.

**Where it appears:**
1. **`session_close_node`** (MIG-04): `KnowledgeGapEngine → NarrativeGenerator → CoachingEngine → SessionClosePipeline → SessionHistory` — five service calls in a single node; all outputs committed to `SessionHistory` at exit.
2. **`reasoner_node` (post-integration)**: `EvaluationSignalWriter → ReasonerService → ObservationExtractor → KnowledgePipeline` — four service calls in a single node; all outputs committed to `InterviewState` fields at exit.
3. **`evaluation_aggregate_node`**: `InterviewEvaluationService.evaluate()` (internally: scoring → signal enrichment → weight normalization → narrative assembly → final decision) — a multi-step pipeline in a single node committing to `state.interview_evaluation`.

**Criteria check:**
- Appears ≥ 3 times: **Yes** (3 confirmed independent instances)
- Services are independent (no cross-calling): **Yes** in all 3 instances
- Output is a single committed artifact or state update: **Yes** in all 3 instances
- Consistent with PAT-06 (thin node, no orchestration logic): **Yes**
- Consistent with PAT-05 (builder-only construction at artifact level): **Yes**

**Methodology status:** Pattern appears in ≥ 3 independent contexts. Satisfies the ≥ 2 occurrence rule for extraction candidacy (Pattern Freeze §Pattern Extraction). **Not yet formalized.** Requires CAR review before extraction.
