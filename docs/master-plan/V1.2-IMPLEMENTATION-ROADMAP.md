## SECTION B — Construction Philosophy

### Frozen Principles

1. **Architecture is finished.** AC-V1.2 is issued. Construction never redesigns architecture.
2. **Every implementation traces to an ADR.** No code may be written without a governing ADR.
3. **Every Epic ends the same way.** Tests → Audit → Freeze → Certification. No exceptions.
4. **Incremental, TDD-first.** No big-bang implementation. Every milestone delivers a testable, auditable increment.
5. **Frozen means frozen.** Once a component is certified frozen, modification requires a new ADR — not a judgment call.

## SECTION C — Construction Roadmap

### EPIC-00 — Language Independence Layer

**Purpose:** Encode Python, JavaScript, and TypeScript as first-class language citizens via a `LanguageConfig` domain object. Eliminate all hardcoded language assumptions from evaluation, generation, and execution paths.

**Governing ADRs:** ADR-019 (LanguageConfig), ADR-027 (LanguageExecutor), ADR-028 (LanguageSelectionPolicy)

**Dependencies:** V1.1 multi-language execution sandbox (complete)

**Expected Deliverables:**
- `LanguageConfig` domain value object
- `LanguageRegistry` (Python, JavaScript, TypeScript registered)
- `LanguagePolicy` artifacts per language
- `LanguageExecutor` abstract interface + per-language dispatch
- `LanguageSelectionPolicy` (mixed-mode 50/50 algorithm)
- Session config extension for language mode
- All hardcoded language references removed from evaluation/generation paths

**Estimated Complexity:** M

**Completion Criteria:**
- All three language modes (Python-only, JS/TS-only, Mixed) produce correct sessions end-to-end
- Language config persists through adaptive navigation
- Zero hardcoded language strings in `services/`, `domain/`, `app/`

**Certification Criteria:**
- `LanguageConfig` schema frozen (`extra=forbid`, `schema_version`)
- `LanguageExecutor` interface frozen
- `LanguageRegistry` frozen
- ADR-019, ADR-027, ADR-028 marked IMPLEMENTED in INDEX

### EPIC-01 — Candidate Knowledge Model (ProfileFeature Activation)

**Purpose:** Freeze the `ProfileFeature` schema and activate the `FeatureEngine`, making the `CandidateProfile` a live, consuming object for downstream components.

**Governing ADRs:** ADR-018 (ProfileFeature schema), ADR-020 (FeatureEngine architecture), ADR-032 (CandidateProfileSnapshot)

**Dependencies:** EPIC-00 (LanguageConfig stable); EPIC-02 read interface contract (ObservationStore interface must be defined, not fully implemented)

**Expected Deliverables:**
- `ProfileFeature` Pydantic schema (version-controlled, `schema_version`)
- `FeatureEngine` orchestrator (replaces direct `CandidateProfileEngine` feature update calls)
- `FeatureUpdater` protocol + `ObservationUpdater` (initial implementation)
- `CandidateProfile.features: dict[str, ProfileFeature]` activated
- `CandidateProfileSnapshot` strategy implemented
- `CandidateProfile` consumed by `NarrativeGenerator` and `ReportBuilder` (interface contract)

**Estimated Complexity:** L

**Completion Criteria:**
- `ProfileFeature` schema frozen and version-controlled in `domain/contracts/`
- `FeatureEngine` is the sole writer of `CandidateProfile.features`
- No other component writes directly to `CandidateProfile.features`
- V1.1 regression: no change to existing `CandidateProfile.dimension_scores` or `CandidateProfile.questions_answered` behaviour

**Certification Criteria:**
- `ProfileFeature` schema certified in `domain/contracts/knowledge/`
- `FeatureEngine` API frozen
- ADR-018, ADR-020, ADR-032 marked IMPLEMENTED

### EPIC-02 — Observation Layer

**Purpose:** Formalise `Observation` lifecycle. Implement `ObservationExtractor` (transforms `EvidenceSignal` → `Observation`), `ObservationStore` (append-only aggregate), and `ObservationType` registry.

**Governing ADRs:** ADR-016 (Observation schema + ObservationExtractor), ADR-017 (ObservationStore lifecycle + temporal semantics)

**Dependencies:** EPIC-01 (ProfileFeature schema stable — `FeatureEngine` must read `ObservationStore`)

**Expected Deliverables:**
- `Observation` Pydantic schema (`frozen=True`, `schema_version`, `ObservationType` field)
- `ObservationType` controlled vocabulary (enum or registry)
- `ObservationExtractor` — sole writer of `ObservationStore`
- `ObservationStore` — append-only, deduplication, temporal ordering, decay-ready interface
- Integration of `ObservationExtractor` into evaluation pipeline (after `EvidenceSignalWriter`)
- `FeatureEngine` reads `ObservationStore` via read-only interface

**Estimated Complexity:** M

**Completion Criteria:**
- Every pattern detector cycle produces correctly typed `Observation` inputs in `ObservationStore`
- `ObservationStore` accumulates correctly across multi-question sessions
- `ObservationExtractor` is the only component that appends to `ObservationStore`
- Duplicate observations (same type + source + question_index) are rejected

**Certification Criteria:**
- `Observation` schema certified in `domain/contracts/knowledge/`
- `ObservationStore` append-only contract verified by tests
- `ObservationExtractor` is sole writer (verified by architecture audit)
- ADR-016, ADR-017 marked IMPLEMENTED

### EPIC-03 — Narrative Generator V2

**Purpose:** Upgrade `NarrativeGenerator` to read `CandidateProfile.features` (ProfileFeatures) and produce coaching text that references specific behavioral patterns and knowledge gaps, not only dimensional scores.

**Governing ADRs:** ADR-023 (NarrativeGenerator profile-feature-aware prompt design), ADR-050 (NarrativeGenerator consumes ProfileFeatures, not detector outputs)

**Dependencies:** EPIC-01 (ProfileFeatures available), EPIC-02 (ObservationStore stable); SessionHistory persistence schema (can be in parallel)

**Expected Deliverables:**
- `NarrativeGenerator` V2 — reads `CandidateProfile.features` + `KnowledgeGap[]`
- Profile-aware prompt templates (`app/prompts/narrative/`)
- `NarrativeSection` structured output schema (`frozen=True`, `schema_version`)
- `NarrativeInsight` — evidence-anchored; references `ProfileFeature` or `Observation`
- `Narrative` aggregate schema — contains `NarrativeSection[]` + `NarrativeInsight[]`
- Behavioural validation suite (≥ 20 scenarios)

**Estimated Complexity:** M

**Completion Criteria:**
- Narrative sections contain at least one ProfileFeature reference when detectors fired
- `NarrativeInsight` always references at least one Observation or ProfileFeature (domain invariant I-15)
- LLM structured output schema validation passes ≥ 99% of sessions
- `NarrativeGenerator` never reads detector outputs or `ObservationStore` directly (ADR-050)

**Certification Criteria:**
- `Narrative`, `NarrativeSection`, `NarrativeInsight` schemas certified
- Behavioural validation suite ≥ 20 scenarios passing
- ADR-023 marked IMPLEMENTED

### EPIC-04 — Coaching Engine

**Purpose:** Implement the Study Recommendations Engine — gap-to-resource mapping producing a prioritised study plan per session.

**Governing ADRs:** ADR-025 (CoachingEngine ranking algorithm), ADR-030 (ResourceLibrary governance), ADR-067 (detector-to-CoachingEngine decoupling)

**Dependencies:** EPIC-03 (Narrative stable); V1.1 `KnowledgeGapEngine` (complete, available)

**Expected Deliverables:**
- `CoachingEngine` — reads `KnowledgeGap[]` + `ProfileFeature[]`; sole producer of `CoachingPlan`
- `CoachingPlan` schema (`frozen=True`, `schema_version`)
- `CoachingAction` schema — maps 1:1 to `KnowledgeGap`
- `StudyRecommendation` schema — topic + gap + resources + estimated study time
- `ResourceLibrary` — curated, version-controlled resource catalog (initial set)
- `LearningObjective` — groups recommendations per knowledge area
- `CoachingEngine` ranking implementation (gap severity × IRS impact)

**Estimated Complexity:** M

**Completion Criteria:**
- 100% of sessions with identified gaps produce a non-empty ranked recommendation list
- Every `CoachingAction` references a specific `KnowledgeGap`
- Zero-gap sessions produce a completion acknowledgement
- `CoachingEngine` never reads raw `EvaluationResults` directly (ADR-067)

**Certification Criteria:**
- `CoachingPlan`, `CoachingAction`, `StudyRecommendation` schemas certified
- `ResourceLibrary` v1.0 tagged and version-controlled
- ADR-025, ADR-030 marked IMPLEMENTED

### EPIC-05 — Evidence Freshness

**Purpose:** Implement observation decay so that early-session signals do not permanently distort the candidate profile. Configurable TTL per observation type, clock-mockable for testing.

**Governing ADRs:** ADR-021 (Knowledge freshness, temporal decay, replay strategy), ADR-017 (ObservationStore lifecycle)

**Dependencies:** EPIC-02 (ObservationStore with decay-ready interface)

**Expected Deliverables:**
- Decay function (configurable shape: linear or step per ADR-021 decision)
- `ObservationStore` expiry logic (TTL per `ObservationType`)
- Mockable clock interface for deterministic tests
- `settings.py` freshness configuration section
- Profile recalculation on each `FeatureEngine` cycle after decay

**Estimated Complexity:** S

**Completion Criteria:**
- Observations beyond configured TTL do not contribute to ProfileFeature computation
- Freshness configuration is centralised in `settings.py`
- All decay tests use mock clock (no wall-clock dependency)
- No regression on active-session behaviour (default TTL > session duration)

**Certification Criteria:**
- Decay function frozen (configurable, not hardcoded)
- TTL configuration certified in `settings.py`
- ADR-021 marked IMPLEMENTED

### EPIC-06 — Calibration Framework

**Purpose:** Systematically validate that scoring constants remain correctly calibrated as the question corpus, language coverage, and candidate profiles evolve. CI gate prevents scoring drift.

**Governing ADRs:** ADR-024 (Calibration Framework CI gate design — Support ADR, must be authored before this Epic begins), ADR-031 (LanguagePolicy governance)

**Dependencies:** EPIC-00 (LanguageConfig); stable evaluation pipeline (V1.1 complete)

**Expected Deliverables:**
- Calibration test suite — synthetic session corpus per (role, seniority, difficulty, language)
- `CalibrationProfile` schema — expected score distributions
- Per-dimension score distribution reports
- CI gate: fails if score distribution drifts beyond threshold
- `CalibrationUpdater` (FeatureEngine updater for calibration-aware features)

**Estimated Complexity:** M

**Completion Criteria:**
- Calibration suite runs in CI without manual intervention
- Score distribution for synthetic sessions is within expected range per difficulty/seniority/language matrix
- Any change to evaluation constants in `evaluation.py` triggers automatic calibration run
- `CalibrationProfile` is a read-only validation artifact (never modifies constants)

**Certification Criteria:**
- CI calibration gate active and passing
- Baseline `CalibrationProfile` v1.0 tagged
- ADR-024, ADR-031 marked IMPLEMENTED

## SECTION E — Construction Workflow

### Per-Milestone Mandatory Workflow

```
Implementation
    ↓
Unit Tests         (≥ 90% coverage; TDD-first)
    ↓
Integration Tests  (pipeline contracts verified end-to-end)
    ↓
Architecture Audit (no boundary violations, no second writers, no hidden coupling)
    ↓
Documentation Update (TDS section, ADR status, contracts registered)
    ↓
Freeze             (API declared frozen; no breaking changes without ADR)
    ↓
Certification      (milestone registered in INDEX; Epic advances)
```

### Responsibilities

| Stage | Responsibility | Authority |
|---|---|---|
| Implementation | Express frozen contracts in code; no architectural decisions | Implementation Baseline rules |
| Unit Tests | Cover all happy paths + boundary conditions; use mock clock where applicable | Quality Gate (≥90% coverage) |
| Integration Tests | Verify single-writer invariants across components; verify pipeline contracts | Merge Gate |
| Architecture Audit | Confirm no boundary violations, no second writers, no circular imports | Audit Gate |
| Documentation Update | TDS current section, ADR status → IMPLEMENTED, contracts registered | Documentation Quality |
| Freeze | Record API surface in INDEX; warn on any future change attempt | Freeze Gate |
| Certification | Update INDEX frozen baseline; register technical debt if any | Certification Gate |

## SECTION G — Risk Register

### Technical Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| `FeatureEngine` becomes a God Object absorbing CandidateProfileEngine logic | Medium | High | ADR-020 defines strict composition model; enforce via architecture audit at E01-M4 |
| `ObservationExtractor` introduces a second path to write EvidenceStore | Low | Critical | Single-writer audit at E02-M5; CI import guard to prevent ObservationStore writes from outside ObservationExtractor |
| LLM structured output failures in NarrativeGenerator V2 | High | Medium | Retry with fallback schema; enforce validation ≥ 99%; schema is narrow and deterministic |
| Decay function produces non-deterministic test behaviour | Medium | Medium | Mockable clock interface (E05-M1); all decay tests use injected clock |
| `ProfileFeature` schema drift between V1.1 detectors and V1.2 consumers | Low | High | Schema frozen in `domain/contracts/knowledge/` with `extra=forbid` before any consumer is implemented |

### Architectural Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| `NarrativeGenerator` bypasses ProfileFeature layer and reads detector outputs directly | Low | High | ADR-050 boundary; architecture audit at E03-M3 |
| `CoachingEngine` reads raw EvaluationResults instead of KnowledgeGaps | Low | High | ADR-067 boundary; import restriction; architecture audit at E04-M3 |
| `SessionHistory` receives a second write after session close | Low | Critical | Write-once enforcement in persistence layer; integration test verifies idempotency |
| New language added without updating `LanguageRegistry` | Medium | Medium | Registry registration is a required step in EPIC-00 extensibility playbook |

### Testing Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Behavioural validation suite (EPIC-03) too narrow to catch narrative quality regression | Medium | Medium | Minimum 20 scenarios covering: no-detector-fired, single-detector, multi-detector, gap+feature combination |
| Calibration corpus (EPIC-06) too small to detect scoring drift | Medium | High | Minimum 50 synthetic sessions per (difficulty × seniority) cell before baseline is locked |
| Integration tests for ObservationStore decay not covering edge cases (TTL boundary) | Medium | Medium | Parametric tests at TTL - 1, TTL, TTL + 1 question intervals |

### Performance Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| `FeatureEngine` full cycle exceeds 20ms budget | Low | Medium | Profile at E01-M4; each FeatureUpdater is individually bounded |
| `ObservationStore` scan on large session (20+ questions) becomes O(n²) | Low | Medium | Enforce indexed lookup; ObservationStore query interface forbids full-scan without filter |

### Integration Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| `ObservationExtractor` pipeline position breaks existing EvaluationSignalWriter flow | Medium | High | Incremental integration at E02-M5; full V1.1 regression must pass before merge |
| SessionHistory schema migration breaks existing V1.1 test fixtures | Medium | Medium | Additive-only migration; fixture backward-compatibility test at E03-M6 |

### Documentation Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| ADR status not updated to IMPLEMENTED after milestone freeze | Medium | Low | Mandatory documentation update in every milestone DoD |
| TDS section becomes stale as implementation diverges from design | Medium | Medium | TDS update is a Freeze Gate requirement; blocked until section is current |
